## 核心架构约束（红线速查）

> 完整约束清单见各架构白皮书，此处仅列最关键条目。

| # | 约束 | 违反后果 |
|---|------|----------|
| 1 | 所有过滤正则必须来自数据库 `filename_clean_regex`，严禁硬编码 | 正则规则与 RegexLab 脱钩，用户设置失效 |
| 2 | 所有 AI 规则（`expert_archive_rules`、`master_router_rules`）必须来自数据库 | AI 行为无法通过 WebUI 热更新 |
| 3 | 所有业务 API 必须通过 `api_router` 挂载，不得绕过全局 JWT 依赖 | 未鉴权端点暴露 |
| 4 | 前端所有 HTTP 调用只能经过 `lib/api.ts`，禁止裸 `fetch` | 401 处理、超时熔断失效 |
| 5 | 6个敏感 API Key 必须通过 `CryptoManager` 加密存储 | 配置文件明文泄露密钥 |
| 6 | 删除操作必须按 `path` 双重清理 `tasks` 和 `media_archive` | 归档记录残留，文件永远无法重新刮削 |
| 7 | `mark_task_as_ignored_and_inherit` 不可拆分（原子操作） | 前端 VHS 特效失效，ignored 任务白板显示 |
| 8 | 前端所有 UI 文本通过 `t(key)` 获取，禁止硬编码字符串 | i18n 切换失效，中英文混显 |
| 9 | `getPosterUrl`（MediaTable）只返回原始物理路径，严禁提前拼接 `/api/v1/public/image?path=` | SecureImage 防套娃误判，请求打向前端 3000 端口 → 404 |
| 10 | 禁止以 `sudo npm run dev` 启动前端开发服务器 | `.next` 缓存文件属主变为 root，代码修改静默失效，极难排查 |
| 11 | `NEXT_PUBLIC_API_BASE` 绝对路径只能存放于 `frontend/.env.development.local`，严禁写入 `.env.local` | 该变量编译期 bake 进 bundle，AIO 镜像构建后局域网设备图片全部失败 |
| 12 | `is_archive` 必须在所有跨端通信中传递 | 后端无法确定操作哪张表，数据写入错误表 |
| 13 | 冷表操作使用 `original_task_id` 作为主键，热表使用 `id` | 冷表数据无法正确更新 |
| 14 | `SecureImage` 只有真正的外部图片才直接渲染，其余必须走 `fetch + Authorization token` | AIO 模式下相对路径 `<img src>` 不携带 token → 401 |
| 15 | `[EXTRACT]` 标记的正则规则仅用于 `extract_season_episode()` 提取，不得加入 `_filter_patterns`（`clean_name()` 不删除） | 集数信息被误删 → AI 丢失集号 → IMDb 查重熔断 → 剧集误标 `ignored` |
| 16 | 保护性前置提取（`extract_season_episode(raw_filename)`）必须在 `clean_name()` 之前调用，结果写回 `task` dict | 晚于去噪执行则 raw_filename 已失去集号信息，锁定机制失效 |

## 开发者指南：双表架构与跨表 ID 契约

### 背景：tasks 热表 vs media_archive 冷表

Neon Crate 采用双表架构管理媒体任务生命周期：

- **热表 `tasks`**：存储未归档任务，主键为自增 `id`
- **冷表 `media_archive`**：存储已归档任务，主键为自增 `id`，同时保存 `original_task_id`（归档前的热表 `id`）

### UNION ALL 虚拟 ID 映射

`stats_repo.get_all_tasks()` 使用 `UNION ALL` 合并两表返回给前端：

```sql
SELECT id, ..., 0 as is_archive FROM tasks
UNION ALL
SELECT original_task_id as id, ..., 1 as is_archive FROM media_archive
```

**关键**：冷表以 `original_task_id` 映射为 `id` 字段对外暴露。前端看到的 `task.id` 对于已归档任务实为 `original_task_id`，而非 `media_archive.id`。

### ID 契约红线

| 操作 | 正确查询键 | 错误查询键 |
|------|-----------|----------|
| 热表写回 | `WHERE id = ?` | — |
| 冷表写回 | `WHERE original_task_id = ?` | `WHERE id = ?` ❌ |

`update_archive_sub_status` 若使用 `WHERE id = ?` 将命中 0 行，导致字幕状态永远停留在 `pending`（次元 ID 隔离故障）。

### 异步字幕智能轮询机制

字幕任务走 FastAPI `background_tasks` 异步链路，API 返回时写库尚未完成（约 4-8 秒延迟）。前端采用指数退避轮询策略：

```
触发后立即刷新（同步字段）
  → 等 3s 再刷新（捕捉初始 pending/triggered）
  → 再等 3s 刷新（捕捉下载中状态）
  → 再等 4s 刷新（捕捉最终 scraped/failed）
  → 检测到终态（scraped/found/failed/missing）立即停止
```

终态集合：`['scraped', 'found', 'failed', 'missing']`，最多轮询 3 次（累计 10 秒），严禁无限循环。
