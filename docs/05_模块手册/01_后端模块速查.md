## app/core/lifespan.py

- **职责**：FastAPI `lifespan` 上下文管理器，管理应用启动/关闭全流程
- 启动序列：日志系统初始化（RotatingFileHandler 10MB×5）→ DatabaseManager 初始化 → 孤儿任务清理（重置上次崩溃遗留的 pending）→ 自动巡逻循环 `cron_scanner_loop()` 启动
- 关闭序列：取消 `cron_scanner_loop` asyncio Task，释放资源
- 孤儿任务清理：进程崩溃后重启时，重置所有卡在 `pending` 状态的历史任务

| 元素 | 说明 |
|------|------|
| `lifespan(app)` | `@asynccontextmanager`，注册到 FastAPI 实例 |
| `cron_scanner_loop()` | asyncio 后台巡逻协程（定时扫描→刮削→字幕）|

## 双表操作职责边界

### TaskRepo（热表操作）

**职责**：管理 `tasks` 热表（新任务、处理中、已处理）

| 方法 | 职责 | 返回值 |
|------|------|--------|
| `insert_task(data)` | 插入新任务，path UNIQUE 防重 | task_id |
| `get_tasks_needing_scrape()` | 获取待刮削任务（双表联查，返回 is_archive 标记）| List[Dict] |
| `get_tasks_needing_subtitles()` | 获取待字幕任务（双表联查 + 7天冷却期）| List[Dict] |
| `update_task_status(id, status, ...)` | 更新任务状态（触发归档流程）| None |
| `update_task_title_year(id, title, year, season)` | 更新元数据+季号 | None |
| `update_any_task_metadata(id, is_archive, ...)` | 通用双表元数据更新（根据 is_archive 选择表）| None |
| `mark_task_as_ignored_and_inherit(id, is_archive, ...)` | 🚨 原子操作，标记 ignored + 继承海报 | bool |
| `check_task_exists_by_path(path)` | 路径去重检查（双表查询）| bool |
| `check_task_exists_by_name(fname, cname)` | 文件名去重检查 | bool |
| `check_media_exists(imdb_id, type, season, ep)` | 防重拦截（精确到季集号）| bool |
| `delete_task(id)` | 双表清理删除 | None |
| `reset_orphan_pending_tasks()` | 孤儿任务救援（超时标记为 failed）| int |

**关键设计**：
- `get_tasks_needing_scrape()` 和 `get_tasks_needing_subtitles()` 都执行双表联查，返回 `is_archive` 标记
- `update_any_task_metadata()` 根据 `is_archive` 参数选择操作哪张表
- `mark_task_as_ignored_and_inherit()` 是原子操作，必须一次性完成（跨表查找海报 + 写入 ignored）

### ArchiveRepo（冷表操作）

**职责**：管理 `media_archive` 冷表（已归档、历史记录）

| 方法 | 职责 | 返回值 |
|------|------|--------|
| `archive_task(task_id, ...)` | 将热表任务流转至冷表 | None |
| `get_archived_data(search, include_ignored)` | 获取冷表数据（将 original_task_id 映射为 id）| List[Dict] |
| `get_archived_task_by_id(original_task_id)` | 按 original_task_id 查询冷表 | Dict |
| `update_archive_sub_status(original_task_id, sub_status)` | 更新冷表字幕状态 | None |
| `delete_archived_task(original_task_id)` | 删除冷表记录 | None |
| `get_sibling_poster(imdb_id, type, season, ep)` | 跨表查找同源海报（冷表优先）| str \| None |

**关键设计**：
- 冷表主键为 `original_task_id`（全局唯一身份证）
- `get_archived_data()` 将 `original_task_id` 映射为 `id` 字段返回前端，链路完全闭环
- `get_sibling_poster()` 先查冷表，再查热表，用于防重拦截时继承海报

### StatsRepo（联合查询）

**职责**：执行双表联合查询，生成统计数据

| 方法 | 职责 | 返回值 |
|------|------|--------|
| `get_all_data(search, include_ignored)` | 获取所有任务（UNION ALL 双表合并）| List[Dict] |
| `get_dashboard_stats()` | 实时统计（pending/completed）| StatsResponse |

**关键设计**：
```python
# stats_repo.py: get_all_data()
# 业务链路：1. 构建双表 UNION ALL 查询 -> 
#          2. 统一使用 original_task_id 作为冷表的对外 ID -> 
#          3. 应用搜索过滤（可选）-> 
#          4. 按创建时间倒序排列 -> 
#          5. 将查询结果转换为字典列表

base_query = f"""
    SELECT id, path, file_name, clean_name, type, status, tmdb_id, imdb_id, title, year, 
           target_path, sub_status, last_sub_check, created_at, poster_path, local_poster_path, 
           season, episode 
    FROM tasks 
    WHERE 1=1 {ignored_clause}
    UNION ALL
    SELECT original_task_id as id, path, file_name, clean_name, type, 'archived' as status, 
           tmdb_id, imdb_id, title, year, target_path, sub_status, NULL as last_sub_check, 
           created_at, poster_path, local_poster_path, season, episode 
    FROM media_archive
    WHERE 1=1
"""
```

# 检查媒体是否已存在（精确到季集号）

| 函数签名 | 返回值 | 职责边界 |
|----------|--------|----------|
| `_step_nfo_shortcut(db, task, file_path, task_id, clean_name)` | `bool` | **Step 1 — NFO 短路拦截**：查找本地 NFO → 解析 TMDB/IMDb ID → 直接归档。成功返回 `True`（主流程短路），失败返回 `False`（降级走正常流程）。绝对禁止调用 AI 和 TMDB。 |

| `_step_tmdb_search_and_dup_check(db, scraper, task, task_id, refined_query, refined_year, refined_type, rename_lang)` | `dict \| None` | **Step 3 — TMDB 搜索与防重拦截**：三梯队降级搜索策略（见下表）→ 精确/宽松匹配 → 获取 IMDb ID → 🛡️ IMDb+S+E 三元组防重熔断：`type='tv' 且 episode=None` 时**跳过**防重检测（进入待定状态，防止缺集号误拦截）；三元组精确命中时调用 `mark_task_as_ignored_and_inherit` 原子操作。返回 `None` 表示已处理（failed/ignored），主流程应 `return False, True`。 |
| `_step_archive_and_metadata(db, task, task_id, file_path, tmdb_api_key, poster_lang, refined_type, tmdb_data)` | `str \| None` | **Step 4 — 就地补录/归档全链路**：检测文件是否来自 library → 就地补录模式（仅写 NFO/海报，不移动文件）或归档全链路（SmartLink + NFO + 海报 + Fanart）→ 路径防穿越安全校验 → 更新 DB。返回 `target_path`，`None` 表示失败。 |

### MediaCleaner

```python
cleaner = MediaCleaner(db_manager=db)

# 结构化提取（保护性前置提取，在 clean_name 之前调用）
season, episode = cleaner.extract_season_episode(filename)
# → 支持 S01E01 / S01 E01 / S01.E01 / Season 1 Episode 1 / 1x01 / EP01 / 第01集 等格式
# → 规则 10 已修复支持空格变体（[Ss](\d{1,2})[\s\._-]*[Ee](\d{1,3})）

# 物理正则去噪（[EXTRACT] 规则隔离：规则 10-15 仅用于提取，不参与删除）
clean = cleaner.clean_name(filename)
# → 去除分辨率、编码、语言标签、制作组等噪声，保留标题
# → 季集号（S01E01 等）不会被删除，防止 AI 收到残缺文件名

# 一次性提取（扫描阶段）
result = cleaner.clean_and_extract(filename)
# → {clean_name, year, season, episode, is_tv, is_ad}
```

**`[EXTRACT]` 规则说明**：`filename_clean_regex` 中规则 10–15 注释含 `[EXTRACT]` 标记，`_load_patterns()` 会跳过这些规则，不加入 `_filter_patterns`。`clean_name()` 因此不会删除集数信息。详见 [原子级识别协议](../01_架构设计/04_全栈逻辑交互拓扑蓝图.md#原子级识别协议)。

### NFO 解析“三层装甲防御”（对第三方脏数据的生存模型）

> **[v1.0.0]** `app/services/metadata/nfo_parser.py: parse_nfo()` 实现，用于 **Step 1 — NFO 短路拦截**（自动刮削轨），将本地 NFO 视为「物理真理」。

| 层级 | 机制 | 目的 | 典型脏数据 |
|------|------|------|------------|
| **第一层：读取容错** | `open(..., encoding="utf-8", errors="replace")` | 避免二进制污染/错误编码导致崩溃，保证“至少读得出来” | 混入非 UTF-8 字节、奇怪 BOM |
| **第二层：生化清洗** | `_sanitize_xml(raw)`：剥 BOM → HTML 实体替换 → 裸 `&` 修复 → XML1.0 控制字符剔除 | 让毒化 XML 重新具备可解析性，最大化进入 ET 严格解析通道 | 未转义 `&`、`&nbsp;`、控制字符 |
| **第三层：正则兜底** | `except ET.ParseError: _regex_fallback(raw)` | 当 XML 彻底碎裂时，仍能抢救 `tmdb_id/imdb_id/title/year` 关键字段，确保短路链路不被“脏数据”拖死 | 截断 XML、非法闭合标签、上游写坏格式 |

**关键结论**：三层防御确保系统面对第三方 “脏 NFO / 脏 XML” 仍具备极强生存能力——**能解析就解析，解析不了也要抢救 ID**，从而稳定触发短路归档，避免无谓的 AI/TMDB 消耗。

**TV 剧集 N+1 雪崩防护（资源存在性短路）**：

| 方法 | 短路条件 | 效果 |
|------|---------|------|
| `generate_nfo()` | `os.path.exists(output_path)` | 跳过 TMDB API + XML 构建 + 磁盘写入 |
| `download_poster()` | `os.path.exists(poster.jpg/png)` | 跳过 TMDB API + 图片下载 |
| `download_fanart()` | `os.path.exists(fanart.jpg/png)` | 跳过 TMDB API + 图片下载 |

同剧集第 2 集起，`tvshow.nfo`、`poster.jpg`、`fanart.jpg` 全部走文件系统短路，TMDB API 调用从 N×3 次降至 3 次（节省 > 95%）。

## TMDB 搜索三梯队降级策略

> **[v1.0.0]** `_step_tmdb_search_and_dup_check()` 中实现，适用于剧集类型。

| 梯队 | 搜索参数 | 触发条件 | 解决场景 |
|------|---------|---------|----------|
| **第一梯队（精确）** | `Title + Year` | 默认首次搜索 | 年份正确时直接命中 |
| **第二梯队（容错，核心武器）** | `Title完整片名 + 无Year` | 第一梯队 0 结果 且 原本有年份参数 | **专杀 AI 年份幻觉**：把年份从搜索参数中物理剥离，解决“庆余年惨案”等年份污染导致的 0 命中 |
| **第三梯队（模糊）** | `截断首词 + 无Year` | 第二梯队仍 0 结果 且 片名含空格 | 片名本身存在集号后缀等极端噪音 |

**关键原则**：第二梯队和第三梯队均不携带年份，彻底隔离 AI 幻觉年份污染。电影类型不触发降级（精确搜索失败直接标记 failed）。

## 创世配置自愈注入（Genesis Config Healing）

> **[v1.0.0]** `ConfigRepo._inject_ai_defaults()` 实现，由 `DatabaseManager.__init__` 在初始化末尾调用。

**机制**：系统启动时绕过 `get_config()` 的内存兜底层，直接读取 `config.json` 原始内容，对以下关键字段执行「缺啥补啥」自愈注入：

| 字段 | 板块 | 若缺失则注入 |
|------|------|-------------|
| `ai_name` | AI | 默认名称 `Neon Crate` |
| `ai_persona` | AI | 完整系统内核机长人格描述 |
| `expert_archive_rules` | AI | 完整归档专家规则（含6字段输出契约）|
| `master_router_rules` | AI | 完整意图路由规则（含所有意图白名单）|
| `filename_clean_regex` | 正则 | 15条工业级去噪正则 |
| `supported_video_exts` | 文件格式 | `.mkv,.mp4,.avi,...` |
| `supported_subtitle_exts` | 文件格式 | `.srt,.ass,.vtt,...` |

**注意事项**：
- 用户已填写的非空值**绝不覆盖**（幂等安全）
- `config.json` 不存在时视为全量空配置，7个字段全部注入
- **不能**用 `get_config()` 判断字段是否为空——`get_config()` 内置 `DEFAULT_CONFIG` 兜底会屏蔽真实空值
