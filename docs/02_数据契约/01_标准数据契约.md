## 一、核心数据流转与 is_archive 契约

```
原始文件名
  ↓ MediaCleaner.clean_and_extract()
  └─ {clean_name, year, season, episode, is_tv, is_ad}
  ↓ ScanEngine 入库
  └─ Task: {path, file_name, clean_name, year, season, episode, is_archive=0, ...}
  ↓ 刮削流程（TMDBAdapter + MetadataManager + SmartLink）
  └─ Task: {title, year, season, tmdb_id, status='archived', is_archive=0/1, ...}
  ↓ 冷表归档（archive_task()）
  └─ media_archive: {original_task_id, ..., is_archive=1}
```

### 🚨 is_archive 致命重要性

**定义**：
- `is_archive=0`：热表 `tasks`（新任务、处理中、已处理）
- `is_archive=1`：冷表 `media_archive`（已归档、历史记录）

**跨端通信中的致命重要性**：

1. **前后端数据契约**：
   ```typescript
   // 前端请求体（RebuildDialog.tsx）
   {
     task_id: number;
     is_archive: boolean;  // ← 必须传递！决定后端操作哪张表
     tmdb_id?: number;
     nuclear_reset: true;
   }
   
   // 后端处理（manual_rebuild）
   if is_archive:
       table = "media_archive"
       pk_field = "original_task_id"
   else:
       table = "tasks"
       pk_field = "id"
   ```

2. **数据库双表寻址**：
   ```python
   # task_repo.py: update_any_task_metadata()
   # 业务链路：1. 根据 is_archive 选择目标表 -> 
   # 2. 构建动态 UPDATE 语句 -> 
   # 3. 执行原子级 UPDATE 并提交事务
   
   if is_archive:
       # ✅ 冷表用 original_task_id 匹配（全局唯一身份证）
       table = "media_archive"
       pk_field = "original_task_id"
   else:
       table = "tasks"
       pk_field = "id"
   ```

3. **前端 UI 路由**：
   ```typescript
   // MediaTable.tsx: 任务行渲染
   // 业务链路：1. 根据 is_archive 决定显示样式 -> 
   // 2. 冷表任务显示为"已归档"（灰色）-> 
   // 3. 热表任务显示为"处理中"（彩色）
   
   const isArchived = task.is_archive === 1;
   const statusColor = isArchived ? 'text-gray-400' : 'text-cyber-cyan';
   ```

4. **防重拦截链路**：
   ```python
   # scrape_task.py: mark_task_as_ignored_and_inherit()
   # 业务链路：1. 跨表检索同源海报路径 -> 
   # 2. 原子写入 ignored 状态 + 继承海报 -> 
   # 3. 若拆分此调用，前端将出现严重破图（VHS 特效丢失）
   
   db.mark_task_as_ignored_and_inherit(
       task_id=task_id,
       is_archive=is_archive,  # ← 决定操作哪张表
       imdb_id=imdb_id,
   )
   ```

**违规后果**：
- ❌ 若前端未传递 `is_archive`，后端无法确定操作哪张表 → 数据写入错误表 → 前端查询不到 → 用户看不到更新
- ❌ 若后端忽略 `is_archive`，冷表数据永远无法更新 → 已归档任务元数据无法补全
- ❌ 若前端显示时混淆 `is_archive`，冷表任务被误认为热表 → UI 状态混乱

## 三、前端类型定义契约与 is_archive 映射

### Task 接口（types/index.ts）

```typescript
export interface Task {
  id: number;
  file_path: string;                    // SRC：原始下载路径
  file_name?: string;
  media_type: 'movie' | 'tv';          // 对应后端 type 字段
  status: 'pending' | 'scraped' | 'failed' | 'archived' | 'ignored';
  is_archive: 0 | 1;                   // ← 🚨 致命重要：0=热表，1=冷表
  tmdb_id?: number | string;
  imdb_id?: string;
  title?: string;
  year?: number | string;              // 宽松类型，前端防御
  poster_path?: string;
  local_poster_path?: string;
  target_path?: string;                // DST：入库后的目标路径
  sub_status?: 'pending' | 'failed' | 'missing' | 'scraped' | 'found';
  season?: number | null;              // 整数类型，非字符串
  episode?: number | null;
  created_at: string;                  // 后端 NULL 兜底为空字符串
}
```

### is_archive 前后端映射表

| 场景 | is_archive | 表名 | 主键字段 | 前端显示 | 操作权限 |
|------|-----------|------|---------|---------|---------|
| 新扫描任务 | 0 | tasks | id | "处理中" | 可编辑、可删除 |
| 处理中任务 | 0 | tasks | id | "处理中" | 可编辑、可删除 |
| 已归档任务 | 0 | tasks | id | "已入库" | 只读、可重建 |
| 历史归档 | 1 | media_archive | original_task_id | "已归档" | 只读、可重建 |
| 重复标记 | 0/1 | tasks/media_archive | id/original_task_id | "已忽略" | 只读 |

### 双轨路径设计（file_path vs target_path）

**前世（SRC）**：`file_path`
- 含义：文件在下载源中的原始路径
- 示例：`/downloads/The.Matrix.1999.1080p.BluRay.x264.mkv`
- 用途：追溯文件来源、重新扫描、调试

**今生（DST）**：`target_path`
- 含义：文件入库后的目标路径（已重命名、已移动）
- 示例：`/media/movies/The Matrix (1999)/The Matrix (1999).mkv`
- 用途：媒体库寻址、播放器调用、元数据关联

**前端 UI 展示**（MediaTable.tsx）：
```typescript
// 🚀 今生：目标入库路径 (置于上方)
{task.target_path && (
  <p className="text-cyber-cyan/30 text-xs truncate mt-1">
    <span className="text-cyber-cyan/50 font-mono mr-1">{t('path_dst')}:</span>
    {task.target_path}
  </p>
)}

// 前世：原始绝对路径 (置于下方)
<p className="text-cyber-cyan/40 text-xs truncate mt-0.5">
  <span className="text-cyber-cyan/60 font-mono mr-1">{t('path_src')}:</span>
  {task.file_path}
</p>
```

**关键约定**：
- `status` 枚举无 `'success'`，严禁使用模糊终态
- `sub_status` 枚举永久移除 `'success'`，合法终态仅为 `'scraped'`（刮削完成）和 `'found'`（本地发现）
- `season` / `episode` 必须为整数，不是字符串
- `created_at` 后端 media_router.py 已在查询层将 NULL 兜底为空字符串，防止前端 `.substring(null)` 崩溃
- **`is_archive` 必须在所有跨端通信中传递**，否则后端无法确定操作哪张表

> 🚨 **[2026-03-15 契约变更]** `sub_status` 中的 `'success'` 已被永久移除。
> 
> **变更原因**：`'success'` 语义模糊，违反 Neon Crate 命名规范（终态必须描述具体完成方式）。
> **替代方案**：
> - `'scraped'` — 通过 API 刮削完成（远程获取）
> - `'found'` — 本地发现已有字幕文件（本地白嫖）
> 
> **进度条影响**：`MediaTable.tsx` 的 `getProgress()` 已同步更新，100% 条件为 `ss === 'scraped' || ss === 'found'`。

## 五、API 数据类型约定与 is_archive 传递

| 字段 | 后端 DB | 前端 | 约定 | is_archive 影响 |
|------|---------|------|------|----------------|
| 媒体类型 | `type` | `media_type` | `'movie'` 或 `'tv'` | 无 |
| 任务状态 | `status` | `status` | 无 `'success'` | 无 |
| 字幕状态 | `sub_status` | `sub_status` | 合法终态：`'scraped'` / `'found'`，永久移除 `'success'` | 无 |
| 季/集号 | INTEGER | number \| null | 整数，非字符串 | 无 |
| 年份 | TEXT | number \| string | 前端宽松兼容 | 无 |
| created_at | TEXT（可NULL）| string（非空）| 后端查询层兜底空字符串 | 无 |
| **is_archive** | **INTEGER** | **0 \| 1** | **🚨 必须传递！决定操作哪张表** | **致命** |
| 主键字段 | `id` / `original_task_id` | `id` | 冷表用 `original_task_id` 映射为 `id` | **致命** |
| 原始路径 | `path` | `file_path` | SRC：下载源路径 | 无 |
| 目标路径 | `target_path` | `target_path` | DST：入库后路径 | 无 |

### is_archive 跨端传递示例

**前端请求（RebuildDialog.tsx）**：

```typescript
const handleNuclearExecute = () => {
```

### manual_rebuild：物理感知护盾（“有 NFO 但无海报”自动补领）

> **[v1.0.0 现状]** `manual_rebuild()` 的“跳过补录（skip_metadata）”不是单纯逻辑判断，而是 **ID 金标准 + 物理存在性** 的组合护盾。

| 条件 | 含义 | 结果 |
|------|------|------|
| NFO 存在 + NFO 内 IMDb ID 与重建目标一致 | 逻辑一致性通过 | 进入“可开启护盾”候选态 |
| 同时海报 `poster.*` 物理存在 | 物理完整 | 允许开启护盾，跳过重复补录 |
| NFO 存在但海报缺失（残缺态） | 物理不完整 | **强制解除护盾**，触发 `refix_poster` 补领 |

**数据库操作（task_repo.py: update_any_task_metadata）**：
```python
# 业务链路：1. 根据 is_archive 选择目标表 -> 
# 2. 构建动态 UPDATE 语句 -> 3. 执行原子级 UPDATE

if is_archive:
    table = "media_archive"
    pk_field = "original_task_id"  # ← 冷表主键
else:
    table = "tasks"
    pk_field = "id"                # ← 热表主键

conn.execute(f"UPDATE {table} SET ... WHERE {pk_field} = ?", params)
```

## 六、刮削任务进度状态契约（V2.0 ScrapeStatusManager）

> **[v1.0.0 新增]** `scrape_all_status` 已从裸字典升级为线程安全的 `ScrapeStatusManager`。`GET /tasks/scrape_all/status` 端点通过 `get_snapshot()` 返回原子快照。

### 状态快照格式（`ScrapeStatusManager.get_snapshot()`）

```json
{
  "is_running": false,
  "last_run_time": 1710000000.0,
  "processed_count": 42,
  "error": null
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `is_running` | `bool` | 当前是否有刮削任务在执行 |
| `last_run_time` | `float \| null` | 上次完成时间（Unix 时间戳），`finish_run()` 原子写入 |
| `processed_count` | `int` | 上次运行处理的任务数，`finish_run()` 原子写入 |
| `error` | `str \| null` | 最近一次错误信息，`set_error()` 原子写入；`set_running(clear_error=True)` 清空 |

**线程安全保证**：所有字段的读写均受 `threading.Lock` 保护，前端轮询 `/status` 端点永远获得一致快照，不存在半写状态。

**合规性约束**：
- 禁止直接操作 `scrape_all_status._state` 字典
- 所有写入必须通过原子方法：`set_running()` / `set_error()` / `finish_run()`
- 只读查询使用 `get_snapshot()` 或 `__getitem__`（均持锁）
