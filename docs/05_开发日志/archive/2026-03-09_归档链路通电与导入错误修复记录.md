## 问题背景

归档流程因 `name 'os' is not defined` 错误崩溃，导致文件无法物理移动到媒体库。这是一个致命的系统级故障，直接阻断了整个归档链路的运行。

### 崩溃现场

```python
# 错误堆栈
NameError: name 'os' is not defined
  File "backend/app/api/v1/endpoints/tasks.py", line 298
    file_ext = os.path.splitext(file_path)[1]
```

### 影响范围

- 所有归档任务无法执行
- 文件无法从下载目录移动到媒体库
- 元数据生成流程被阻断
- 用户无法正常使用归档功能

### 第二步：加固数据库配置获取

**文件**: `backend/app/infra/database/db_manager.py`

**问题**: `get_agent_config` 方法虽然声明了 3 个参数，但内部逻辑不支持直接从 settings 中获取配置项。

**修改内容**:

```python
# 修改前
def get_agent_config(self, key: str = None, default: Any = None) -> Any:
    settings = self.get_all_config().get("settings", {})
    
    agent_config = {
        "provider": settings.get("llm_provider", "cloud"),
        # ... 其他配置
    }
    
    if key:
        return agent_config.get(key, default)
    
    return agent_config

# 修改后
def get_agent_config(self, key: str = None, default: Any = None) -> Any:
    config = self.get_all_config().get("settings", {})
    
    # 如果指定了 key，直接从 settings 中获取
    if key:
        return config.get(key, default)
    
    # 否则返回完整的 agent_config 字典
    agent_config = {
        "provider": config.get("llm_provider", "cloud"),
        # ... 其他配置
    }
    
    return agent_config
```

**修复逻辑**:
1. 当调用 `get_agent_config("tmdb_api_key")` 时，直接从 `settings` 中获取
2. 当调用 `get_agent_config()` 时，返回完整的 agent_config 字典
3. 支持 `get_agent_config("key", "default_value")` 三参数调用

**影响**: 确保配置获取逻辑的健壮性，避免因配置读取失败导致的归档流程崩溃。

### 第三步：归档路径安全检查

**文件**: `backend/app/api/v1/endpoints/tasks.py`

**问题**: Windows 环境下路径解析可能出现盘符问题，需要在执行 `os.makedirs` 前进行物理验证。

**修改内容**:

```python
# 修改前
try:
    library_root = db.get_active_library_path(refined_type)
    
    # 构建目标路径
    file_ext = os.path.splitext(file_path)[1]
    # ...
    
    # 执行智能链接
    success, link_type = SmartLink.create_link(file_path, target_path)

# 修改后
try:
    library_root = db.get_active_library_path(refined_type)
    logger.info(f"[ORG] 媒体库根路径: {library_root}")
    
    # 构建目标路径
    file_ext = os.path.splitext(file_path)[1]
    # ...
    
    # 物理路径安全检查（Windows 盘符验证）
    logger.info(f"[ORG] 目标路径: {target_path}")
    logger.info(f"[ORG] 源文件路径: {file_path}")
    
    # 确保目标目录存在
    logger.info(f"[ORG] 创建目标目录: {target_dir}")
    os.makedirs(target_dir, exist_ok=True)
    
    # 执行智能链接
    success, link_type = SmartLink.create_link(file_path, target_path)
    
    if success:
        logger.info(f"[ORG] [SUCCESS] 归档成功 ({link_type}): {target_path}")
    else:
        logger.error(f"[ORG] [FAIL] 归档失败: {link_type}")
```

**新增日志**:
1. `[ORG] 媒体库根路径` - 打印媒体库根目录
2. `[ORG] 目标路径` - 打印完整目标路径
3. `[ORG] 源文件路径` - 打印源文件路径
4. `[ORG] 创建目标目录` - 打印即将创建的目录
5. `[ORG] [SUCCESS]` - 归档成功标记
6. `[ORG] [FAIL]` - 归档失败标记

**影响**: 
- 提供完整的归档路径审计日志
- 便于排查 Windows 路径问题
- 符合安全红线要求（显示 [ORG] [SUCCESS] 或 [ORG] [FAIL]）

### 第四步：剧集匹配失败二次搜索

**文件**: `backend/app/api/v1/endpoints/tasks.py`

**问题**: AI 提炼的 query 可能包含单集名称（如 "Breaking Bad Pilot"），导致 TMDB 搜索失败。

**修改内容**:

```python
# 使用 AI 提炼后的查询词搜索 TMDB
if refined_type == "movie":
    results = scraper.search_movie(
        query=refined_query,
        year=refined_year
    )
else:
    results = scraper.search_tv(
        query=refined_query,
        year=refined_year
    )

# 新增：剧集匹配失败时的二次搜索逻辑
if (not results or len(results) == 0) and refined_type == "tv":
    # 尝试剥离第一个空格后的所有内容（如 "Breaking Bad Pilot" -> "Breaking Bad"）
    if " " in refined_query:
        fallback_query = refined_query.split(" ")[0]
        logger.info(f"[TMDB] 剧集匹配失败，尝试二次搜索: '{fallback_query}'")
        results = scraper.search_tv(
            query=fallback_query,
            year=refined_year
        )

if results and len(results) > 0:
    # 处理匹配结果
```

**修复逻辑**:
1. 检测剧集搜索是否失败（`results` 为空）
2. 如果 query 包含空格，提取第一个词作为 fallback_query
3. 使用 fallback_query 进行二次搜索
4. 记录二次搜索日志

**示例**:
- 原始 query: "Breaking Bad Pilot"
- 一次搜索失败
- 二次搜索: "Breaking Bad"
- 匹配成功

**影响**: 提升剧集匹配成功率，解决 AI prompt 过宽导致的匹配失败问题。

## 技术细节

### Windows 环境特殊处理

根据 `docs/03_开发日志/Windows环境避坑指南.md` 的要求：

1. **编码安全**: 所有文件操作强制使用 `encoding="utf-8"`
2. **路径处理**: 使用 `os.path` 处理路径，避免硬编码 `/` 开头的绝对路径
3. **日志输出**: 严禁使用 Emoji，使用纯 ASCII 字符（如 `[SUCCESS]` 代替 ✅）

### 归档日志规范

所有归档操作必须显示以下标记：

- `[ORG] [SUCCESS]` - 归档成功
- `[ORG] [FAIL]` - 归档失败
- `[ORG]` - 归档流程日志
