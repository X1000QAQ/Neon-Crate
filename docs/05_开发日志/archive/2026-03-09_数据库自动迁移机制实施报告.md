## 问题描述

### 致命错误现场

物理扫描已发现文件，但在入库时崩溃，报错：

```
table tasks has no column named season
```

### 根本原因

现存数据库 schema 版本过旧，缺少以下新增字段：

- `clean_name` (TEXT): 清洗后的媒体名称
- `season` (INTEGER): 季数（剧集专用）
- `episode` (INTEGER): 集数（剧集专用）
- `poster_path` (TEXT): TMDB 海报 URL
- `local_poster_path` (TEXT): 本地海报路径

### 业务影响

- 物理扫描无法完成入库
- 用户无法使用系统核心功能
- 升级后系统完全不可用

### 核心设计：数据库自动迁移机制

实现 Schema Migration 机制，确保旧版本数据库自动对齐新字段。

#### 实现位置

`backend/app/infra/database/db_manager.py`

#### 关键代码

**步骤 1**: 新增 `_migrate_database()` 方法

```python
def _migrate_database(self):
    """
    数据库自动迁移机制
    
    核心功能：
    - 检测现存数据库 schema 版本
    - 自动添加缺失的字段
    - 确保升级后字段自动对齐
    
    迁移字段清单：
    - clean_name: 清洗后的媒体名称
    - season: 季数（剧集专用）
    - episode: 集数（剧集专用）
    - poster_path: TMDB 海报 URL
    - local_poster_path: 本地海报路径
    """
    with self.db_lock:
        conn = sqlite3.connect(self.db_path)
        conn.execute("PRAGMA journal_mode=WAL")
        
        # 获取当前表的所有列名
        cursor = conn.execute("PRAGMA table_info(tasks)")
        existing_columns = [row[1] for row in cursor.fetchall()]
        
        # 需要确保存在的字段列表
        required_columns = {
            "clean_name": "TEXT",
            "season": "INTEGER",
            "episode": "INTEGER",
            "poster_path": "TEXT",
            "local_poster_path": "TEXT"
        }
        
        # 逐个检查并添加缺失字段
        for col, col_type in required_columns.items():
            if col not in existing_columns:
                print(f"[INFO] [DB] 正在自动升级数据库：新增字段 {col}")
                conn.execute(f"ALTER TABLE tasks ADD COLUMN {col} {col_type}")
        
        conn.commit()
        conn.close()
```

## 技术细节

### 迁移策略

**检测机制**: 使用 SQLite 的 `PRAGMA table_info(tasks)` 获取现存字段列表

**添加逻辑**: 逐个检查必需字段，缺失则执行 `ALTER TABLE tasks ADD COLUMN`

**幂等性保证**: 多次执行不会重复添加字段

**原子性保证**: 所有字段添加在同一个事务中完成

### 执行顺序

系统启动时的初始化顺序：

1. `_init_database()` - 创建表结构（如果不存在）
2. `_migrate_database()` - 自动迁移 Schema（关键步骤）
3. `_migrate_sensitive_keys()` - 迁移明文密钥到加密存储
4. `_inject_ai_defaults()` - 注入 AI 规则默认值

## 第二步：清理残留逻辑

### 问题

扫描任务异常时，`scan_status["is_running"]` 未重置为 `False`，导致前端按钮一直卡在"扫描中"状态。

### 解决方案

修改 `backend/app/api/v1/endpoints/tasks.py` 中的 `perform_scan_task()` 函数：

```python
async def perform_scan_task():
    """执行物理扫描任务的后台函数"""
    global scan_status
    
    try:
        scan_status["is_running"] = True
        scan_status["error"] = None
        
        # ... 扫描逻辑
        
    except Exception as e:
        # 确保异常时重置运行状态，防止按钮卡死
        scan_status["is_running"] = False
        scan_status["error"] = str(e)
        print(f"[ERROR] [API] 扫描任务执行失败: {str(e)}")
        logger.error(f"[API] 扫描任务执行失败: {str(e)}", exc_info=True)
```

**关键改动**:

- 异常捕获时强制重置 `scan_status["is_running"] = False`
- 添加 `exc_info=True` 参数，记录完整堆栈信息
- 日志级别从 `[DEBUG]` 提升为 `[ERROR]`

## 技术亮点

### 1. 零停机升级

用户无需手动备份或迁移数据，系统启动时自动完成 Schema 升级。

### 2. 幂等性保证

多次执行迁移逻辑不会重复添加字段，确保系统稳定性。

### 3. 向后兼容

旧版本数据不受影响，仅添加新字段，不修改现有数据。

### 4. 原子性保证

所有字段添加在同一个事务中完成，要么全部成功，要么全部失败。

### 5. 清晰的日志输出

每个字段添加都有明确的日志记录，便于问题排查。

## 经验总结

### 数据库设计原则

**原则 1**: 新增字段必须允许 NULL 或提供默认值

- 确保旧数据不受影响
- 避免 `NOT NULL` 约束导致迁移失败

**原则 2**: 使用 `ALTER TABLE ADD COLUMN` 而非重建表

- SQLite 不支持 `ALTER TABLE MODIFY COLUMN`
- 重建表需要复杂的数据迁移逻辑

**原则 3**: 迁移逻辑必须幂等

- 多次执行不会产生副作用
- 使用 `PRAGMA table_info` 检测字段是否存在

### Windows 环境注意事项

**注意 1**: 日志输出严禁使用 Emoji

- 使用 `[INFO]`, `[ERROR]` 等 ASCII 标记
- 避免 `UnicodeEncodeError` 崩溃

**注意 2**: 文件编码强制使用 UTF-8

- 所有文件读写必须指定 `encoding="utf-8"`
- 避免 GBK 编码问题

**注意 3**: 异常处理必须重置状态

- 后台任务异常时必须重置 `is_running` 标志
- 防止前端按钮卡死

## 后续优化建议

### 建议 1：版本号管理

在数据库中增加 `schema_version` 表，记录当前 Schema 版本号。

```sql
CREATE TABLE IF NOT EXISTS schema_version (
    version INTEGER PRIMARY KEY,
    applied_at TEXT DEFAULT CURRENT_TIMESTAMP
);
```

### 建议 2：迁移脚本化

将迁移逻辑拆分为独立的迁移脚本，按版本号顺序执行。

```python
migrations = [
    ("v2.1", migrate_v2_1),
    ("v2.2", migrate_v2_2),
]
```

### 建议 3：回滚机制

提供数据库备份和回滚功能，确保迁移失败时可以恢复。

```python
def backup_database():
    shutil.copy(db_path, f"{db_path}.backup")
```
