# 极简纯净链路与类型强制校准

**日期**: 2026-03-09  
**版本**: V1.0  
**作者**: 首席系统架构师

## 一、任务背景

### 问题诊断

1. **前端网络海报兜底浪费 API 请求**
   - 前端 `getPosterUrl` 函数存在 TMDB 在线海报兜底逻辑
   - 每次渲染都会尝试请求 `image.tmdb.org`，浪费带宽和 API 配额
   - 违背本地优先原则，增加不必要的网络依赖

2. **剧集 TMDB 跳转链接错误**
   - 物理扫描阶段可能误判媒体类型（如将剧集识别为电影）
   - AI 刮削阶段虽然校准了类型，但未同步更新数据库 `type` 字段
   - 导致前端根据错误的 `type` 生成 TMDB 链接，剧集跳转到电影页面

### 解决方案

1. 移除前端所有 TMDB 在线海报请求逻辑，强制使用本地海报
2. 在后端刮削成功后，强制校准数据库 `type` 字段为 AI 确认的类型

## 二、实施细节

### 2.1 前端海报逻辑极简化

**文件**: `frontend/components/media/MediaWall.tsx`

**修改前**:
```typescript
const getPosterUrl = (task: Task): string => {
  if (task.status === 'archived' && task.target_path) {
    // 复杂的路径兼容逻辑...
    return `/api/v1/assets/${dirPath}/poster.jpg`;
  }
  
  // 兜底方案：TMDB 在线海报
  if (task.poster_path) {
    return `https://image.tmdb.org/t/p/w500${task.poster_path}`;
  }
  return '/placeholder-poster.jpg';
};
```

**修改后**:
```typescript
const getPosterUrl = (task: Task): string => {
  if (task.status === 'archived' && task.target_path) {
    // 将 /storage/Movies/Dune/Dune.mkv 转换为 /api/v1/assets/Movies/Dune/poster.jpg
    const dirPath = task.target_path.replace(/^[\/\\]storage[\/\\]/, '').split(/[/\\]/).slice(0, -1).join('/');
    return `/api/v1/assets/${dirPath}/poster.jpg`;
  }
  return '/placeholder-poster.jpg'; // 去掉所有请求 image.tmdb.org 的逻辑
};
```

**核心变更**:
- 删除所有 `https://image.tmdb.org` 请求逻辑
- 删除 `task.poster_path` 兜底分支
- 简化路径处理逻辑，仅保留核心路由拼装

### 2.2 后端类型强制校准

**文件**: `backend/app/api/v1/endpoints/tasks.py`

**修改位置**: `perform_scrape_all_task` 函数中，TMDB 匹配成功后的数据库更新逻辑

**新增代码**:
```python
# 更新数据库
db.update_task_status(
    task_id=task_id,
    status="archived",
    tmdb_id=str(tmdb_id),
    imdb_id=imdb_id if imdb_id else "",
    target_path=target_path,
    local_poster_path=local_poster_path
)
db.update_task_title_year(
    task_id=task_id,
    title=title,
    year=year
)

# 强制校准任务类型：将 AI 确认的类型写入数据库
try:
    with db.db_lock:
        import sqlite3
        conn = sqlite3.connect(db.db_path)
        conn.execute("UPDATE tasks SET type = ? WHERE id = ?", (refined_type, task_id))
        conn.commit()
        conn.close()
    logger.info(f"[TMDB] 已校准任务 {task_id} 的媒体类型为: {refined_type}")
except Exception as type_err:
    logger.error(f"[TMDB] 类型校准失败: {type_err}")

success_count += 1
```

**核心逻辑**:
1. 在 TMDB 匹配成功后，立即执行类型校准
2. 使用 `sqlite3` 直接执行 SQL，绕过 ORM 缓存
3. 将 AI 提炼并验证成功的 `refined_type` 强制写入 `tasks.type` 字段
4. 确保前端读取到的类型与 TMDB 数据一致

## 三、技术要点

### 3.1 为什么使用直接 SQL 而非 ORM

**原因**:
- `db.update_task_status` 等方法可能不包含 `type` 字段更新
- 直接 SQL 确保强制更新，避免 ORM 缓存干扰
- 使用 `db.db_lock` 确保线程安全

### 3.2 类型校准时机

**关键节点**: TMDB 匹配成功后，`db.update_task_status` 调用之后

**为什么不在 AI 阶段校准**:
- AI 阶段仅提炼类型，未验证 TMDB 匹配结果
- 只有 TMDB 匹配成功后，才能确认类型正确
- 避免误判导致的类型污染

### 3.3 前端海报加载流程

**新流程**:
1. 检查 `task.status === 'archived'` 且 `task.target_path` 存在
2. 从 `target_path` 提取目录路径
3. 拼装本地海报路由 `/api/v1/assets/{dirPath}/poster.jpg`
4. 如果海报不存在，浏览器自动显示占位图

**优势**:
- 零网络请求，纯本地渲染
- 避免 TMDB API 限流
- 提升页面加载速度
