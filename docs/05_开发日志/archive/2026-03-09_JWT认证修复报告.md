## 一、问题描述

### 故障现象
- 前端点击"重置为默认值"按钮时，请求 `POST /api/v1/tasks/settings/reset` 返回 403 Forbidden
- 影响范围：PersonaSettings 和 RegexLab 两个组件的重置功能

### 根本原因
前端组件直接使用原生 `fetch` 发起请求，未携带 JWT Token 认证头，导致后端拒绝访问。

**问题代码示例**:
```typescript
// PersonaSettings.tsx - 错误写法
const response = await fetch('/api/v1/tasks/settings/reset', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ target: 'ai' })
});
```

---

## 二、解决方案

### 架构原则
所有需要认证的 API 请求必须通过 `api.ts` 中的 `secureFetch` 包装器，确保自动携带 Authorization 令牌。

### 实施步骤

#### 第一步：封装安全的 API 请求

**修改文件**: `frontend/lib/api.ts`

**新增方法**:
```typescript
async resetSettings(target: 'ai' | 'regex'): Promise<{ success: boolean; message: string }> {
  const res = await secureFetch(`${API_BASE}/tasks/settings/reset`, {
    method: 'POST',
    headers: getHeaders(),
    body: JSON.stringify({ target }),
  });
  if (!res.ok) throw new Error('Failed to reset settings');
  return res.json();
}
```

**关键机制**:
- 使用 `secureFetch` 包装请求，自动处理 401 跳转
- 使用 `getHeaders()` 自动注入 `Authorization: Bearer <token>`
- 统一错误处理和类型定义

## 三、技术细节

### secureFetch 工作原理

```typescript
async function secureFetch(url: string, options?: RequestInit): Promise<Response> {
  const res = await fetch(url, options);
  
  if (res.status === 401) {
    window.location.href = '/';
    throw new Error('Unauthorized');
  }
  
  return res;
}
```

**功能**:
1. 包装原生 fetch
2. 自动检测 401 状态码
3. 未授权时自动跳转到登录页
4. 抛出明确的错误信息

### getHeaders 工作原理

```typescript
function getHeaders(includeAuth: boolean = true): HeadersInit {
  const headers: HeadersInit = {
    'Content-Type': 'application/json',
  };
  
  if (includeAuth) {
    const token = getAuthToken();
    if (token) {
      headers['Authorization'] = `Bearer ${token}`;
    }
  }
  
  return headers;
}
```

**功能**:
1. 自动添加 `Content-Type: application/json`
2. 从 localStorage 读取 JWT Token
3. 自动注入 `Authorization` 头
4. 支持可选的认证头控制

## 六、架构改进

### 改进前
```
前端组件 -> 原生 fetch -> 后端 API
           (缺少 Token)    (403 Forbidden)
```

### 改进后
```
前端组件 -> api.resetSettings() -> secureFetch -> 后端 API
           (统一封装)            (自动注入 Token)  (200 OK)
```

### 优势
1. **安全性**: 所有请求自动携带认证信息
2. **可维护性**: 认证逻辑集中管理，修改一处即可
3. **一致性**: 所有 API 调用遵循统一模式
4. **健壮性**: 统一的错误处理和 401 跳转逻辑
