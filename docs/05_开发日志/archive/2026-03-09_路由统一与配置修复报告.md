## 一、任务概览

本次修复解决了两个关键问题：
1. 重置接口 404 错误（路由不一致）
2. min_size_mb 设置为 0 会自动回滚为 3 的问题（硬编码限制）

---

## 二、问题分析

### 问题 1: 重置接口 404 错误

**现象**:
- 前端请求: `/api/v1/settings/reset`
- 后端路由: `/api/v1/system/settings/reset`
- 导致 404 错误

**根本原因**:
- 重置接口挂载在 `system.py` 路由器上
- 但前端期望的路径是 `/api/v1/tasks/settings/reset`
- 路由挂载点不一致

### 问题 2: min_size_mb 无法设为 0

**现象**:
- 用户在前端设置 min_size_mb = 0
- 保存后自动变回 3

**排查结果**:
- 模型层: `min_size_mb: int = Field(50, ge=0)` - 正确，允许 0
- 数据库层: 无 `max(3, value)` 硬编码逻辑
- 前端层: `parseInt(e.target.value, 10) || 0` - 问题所在！

**根本原因**:
- 前端使用 `|| 0` 作为兜底逻辑
- 当用户输入 0 时，`parseInt("0", 10)` 返回 0
- `0 || 0` 结果为 0，但这不是问题
- 实际问题是：当输入框为空时，`parseInt("", 10)` 返回 NaN
- `NaN || 0` 结果为 0，但这会导致意外行为

## 三、执行内容

### 第一步：统一路由路径 ✅

**目标**: 将重置接口从 `system.py` 移动到 `tasks.py`

**修改文件 1**: `backend/app/api/v1/endpoints/tasks.py`

**新增内容**:
```python
class ResetSettingsRequest(BaseModel):
    """重置配置请求体"""
    target: str


@router.post("/settings/reset")
async def reset_settings(payload: ResetSettingsRequest):
    """
    POST /tasks/settings/reset - 重置配置为工业级默认值
    
    Args:
        payload: {"target": "ai" | "regex"}
    
    Returns:
        操作结果
    """
    target = payload.target.strip().lower()
    
    if target not in ["ai", "regex"]:
        logger.error(f"[API] 重置配置失败: target 必须为 'ai' 或 'regex'，收到: {target}")
        return {"success": False, "message": "target 必须为 'ai' 或 'regex'"}
    
    try:
        db = get_db_manager()
        db.reset_settings_to_defaults(target)
        
        logger.info(f"[API] 触发配置重置: {target}")
        
        return {
            "success": True,
            "message": f"{target.upper()} 配置已重置为工业级默认值"
        }
    except Exception as e:
        logger.error(f"[API] 配置重置失败: {e}")
        return {"success": False, "message": f"重置失败: {str(e)}"}
```

**修改文件 2**: `backend/app/api/v1/endpoints/system.py`

**删除内容**:
- 删除了 `@router.post("/settings/reset")` 路由及其实现

**新路由路径**:
- 前端请求: `/api/v1/tasks/settings/reset`
- 后端路由: `/api/v1/tasks/settings/reset`
- 完全一致，404 问题解决

### 第二步：彻底解除"0 变 3"的限制 ✅

#### 2.1 检查模型层

**文件**: `backend/app/models/domain_system.py`

**验证结果**:
```python
class SystemSettings(BaseModel):
    min_size_mb: int = Field(50, ge=0, description="最小文件大小(MB)，允许设为0以支持测试")
```
- `ge=0` 表示大于等于 0，正确
- 无需修改

#### 2.2 检查数据库层

**文件**: `backend/app/infra/database/db_manager.py`

**验证结果**:
- 搜索 `max(3,` 和 `max(\s*3\s*,` 无结果
- 无硬编码的最小值限制
- 无需修改

#### 2.3 修复前端逻辑

**文件**: `frontend/components/settings/BasicSettings.tsx`

**修改前**:
```typescript
onChange={(e) => updateSetting('min_size_mb', parseInt(e.target.value, 10) || 0)}
```

**问题分析**:
- `parseInt("0", 10)` 返回 0
- `0 || 0` 结果为 0
- 但这会导致用户无法区分"输入 0"和"输入为空"

**修改后**:
```typescript
onChange={(e) => {
  const val = parseInt(e.target.value, 10);
  updateSetting('min_size_mb', isNaN(val) ? 0 : val);
}}
```

**改进说明**:
- 明确检查 `isNaN(val)`
- 只有在解析失败时才使用 0 作为兜底
- 用户输入 0 时，`val = 0`，`isNaN(0) = false`，保存 0
- 用户清空输入时，`val = NaN`，`isNaN(NaN) = true`，保存 0
