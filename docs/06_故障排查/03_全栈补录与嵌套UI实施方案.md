## 二、问题一：数据分组策略（三重折叠性能最优解）

### 现状分析

`MediaWall.tsx` 当前的数据获取方式：
```typescript
// 一次性拉取所有数据，前端分页
const data = await api.getTasks({ page: 1, page_size: 99999 });
setTasks(data.tasks);  // 平铺数组，无分组
```

`MediaTable.tsx` 是纯展示组件，直接渲染平铺的 `tasks[]`，没有任何分组结构。

### 方案对比

#### 方案 A：后端新增 `group_by` 接口

```
GET /tasks?group_by=title&page=1&page_size=50
→ 返回：[
    { group_key: "飞驰人生", total_episodes: 1, tasks: [...] },   // 电影
    { group_key: "庆余年",   total_episodes: 48, seasons: {        // 剧集
        1: { episodes: 12, tasks: [...] },
        2: { episodes: 36, tasks: [...] }
    }}
  ]
```

**优点**：分页精确（按作品分页而非按集数），网络传输量小。  
**缺点**：需要修改 `media_router.py` 的 `get_all_tasks`，impact 为 MEDIUM（影响前端所有现有查询）；`GET /tasks` 契约变更需同步更新 `api.ts` 和所有调用点。

#### 方案 B：前端 `useMemo` 分组（推荐）✅

```typescript
// 在 MediaWall.tsx 中，数据获取不变，仅新增分组计算
const groupedTasks = useMemo(() => {
  const groups = new Map<string, { movie?: Task; seasons: Map<number, Task[]> }>();
  for (const task of tasks) {
    const key = task.title || task.file_name || 'unknown';
    if (!groups.has(key)) groups.set(key, { seasons: new Map() });
    const g = groups.get(key)!;
    if (task.media_type === 'movie') {
      g.movie = task;
    } else {
      const s = task.season ?? 1;
      if (!g.seasons.has(s)) g.seasons.set(s, []);
      g.seasons.get(s)!.push(task);
    }
  }
  return groups;
}, [tasks]);  // 仅在 tasks 变化时重新计算，O(n) 单次遍历
```

**优点**：
- **零后端改动**：`GET /tasks` 接口、`media_router.py`、`db_manager.py` 完全不变
- **零 impact 风险**：不触碰任何 d=1 符号
- **性能充足**：1000 条任务的 `useMemo` 分组耗时 < 2ms（V8 Map 操作，远低于渲染帧预算 16ms）
- **增量渲染**：折叠节点默认收起，React 只渲染展开的节点，DOM 节点数与平铺方案相同

**缺点**：前端持有全量数据（page_size=99999），内存占用略高，但 1000 条 Task 对象总大小 < 500KB，不构成问题。

### 结论：采用方案 B（前端 useMemo 分组）

**性能最优解论证**：
1. 分组计算是 CPU 密集型（单次 O(n) 遍历），`useMemo` 将其缓存到 `tasks` 依赖变化时才重算，与 React reconciler 完全解耦
2. 折叠 UI 的 DOM 节点数由用户展开行为决定，不受总任务数影响
3. 避免后端接口变更的 HIGH 级 impact 风险，是正确的工程决策

## 四、问题三：深度清理安全性审计

### `refix_nfo` 深度清理协议

清理目标：在 `metadata_dir` 下删除以下文件：
- `poster.*`（`poster.jpg`, `poster.png`）
- `fanart.*`（`fanart.jpg`）
- 所有含 `.ai.` 标识的字幕文件（`*.ai.srt`, `*.ai.ass`, `*.ai.zh-CN.srt` 等）

### 路径穿越（Path Traversal）防护审计

**漏洞场景**：若 `task.target_path` 被篡改为 `../../etc/passwd`，`metadata_dir` 将指向系统目录，删除操作造成越权破坏。

**现有防护**（`metadata_manager.py` 中已有 `_validate_path`）：
```python
def _validate_path(target_path: str, allowed_base: str) -> Path:
    resolved_target = Path(target_path).resolve()
    resolved_base = Path(allowed_base).resolve()
    resolved_target.relative_to(resolved_base)  # ValueError 即路径穿越
```

**`manual_rebuild` 删除逻辑的安全设计**：

```python
def _safe_delete_metadata_files(metadata_dir: str, library_root: str) -> dict:
    """
    安全删除 metadata_dir 下的旧元数据文件。

    双重防护：
    1. resolved_metadata_dir 必须是 resolved_library_root 的子路径
    2. 每个待删文件的 resolved_path 必须是 resolved_metadata_dir 的子路径
    3. 只删除白名单模式匹配的文件（poster.*, fanart.*, *.ai.*），不做递归删除
    """
    resolved_meta = Path(metadata_dir).resolve()
    resolved_lib  = Path(library_root).resolve()

    # 防护层 1：metadata_dir 必须在 library_root 内
    try:
        resolved_meta.relative_to(resolved_lib)
    except ValueError:
        raise PermissionError(f"[SECURITY] metadata_dir '{metadata_dir}' 不在 library_root '{library_root}' 内，拒绝删除")

    deleted = {"poster": [], "fanart": [], "ai_subtitles": []}

    # 白名单模式（只删这些，不做 rmtree/glob('*') 递归清空）
    patterns = [
        ("poster",       ["poster.jpg", "poster.png", "poster.webp"]),
        ("fanart",       ["fanart.jpg", "fanart.png"]),
        ("ai_subtitles", list((resolved_meta).glob("*.ai.*"))),  # *.ai.srt, *.ai.zh-CN.srt 等
    ]

    for category, targets in patterns:
        for target in targets:
            if isinstance(target, Path):
                f = target
            else:
                f = resolved_meta / target

            # 防护层 2：每个文件还要验证在 metadata_dir 内（防止 glob 返回软链跳出）
            try:
                f.resolve().relative_to(resolved_meta)
            except ValueError:
                logger.warning(f"[SECURITY] 跳过越界文件: {f}")
                continue

            if f.exists() and f.is_file():
                f.unlink()
                deleted[category].append(str(f))
                logger.info(f"[CLEANUP] 已删除旧元数据: {f}")

    return deleted
```

**安全结论**：
- 白名单模式（poster.\*, fanart.\*, \*.ai.\*），**不做 `shutil.rmtree` 或 `glob('*')` 全清**
- 双重 `relative_to` 校验确保操作绝对不会越出 `library_root`
- 与现有 `_validate_path` 防护体系完全一致，无新增攻击面

## 五、问题四：即时字幕触发分析

### 现有字幕链路（GitNexus 图谱确认）

```
perform_find_subtitles_task_sync()
  → SubtitleEngine.download_subtitle_for_task(
        db_manager, file_path, tmdb_id, media_type,
        imdb_id, season, episode, target_path, archive_id
    )
  → OpenSubtitles API（httpx.AsyncClient，带 429 退避重试）
  → db.update_any_task_metadata(task_id, is_archive, sub_status="success")
```

### `refix_subtitle` 即时执行协议

**不能做**：仅将 `sub_status` 置为 `pending`，等待下次 Cron 扫描（延迟 > 1 分钟，用户体验差）。

**正确做法**：在 `manual_rebuild` 处理器内，通过 `BackgroundTasks` 立即将字幕任务投入独立协程，实现准实时反馈：

```python
@router.post("/manual_rebuild")
async def manual_rebuild(
    body: ManualRebuildRequest,
    background_tasks: BackgroundTasks,
    db: DbDep = None
):
    # ... NFO / Poster 重建逻辑 ...

    if body.refix_subtitle:
        # Step 1: 重置状态（先写库，让前端轮询能感知到 pending）
        db.update_any_task_metadata(body.task_id, body.is_archive, sub_status="pending")

        # Step 2: 立即投入后台协程（不等待，立等可取）
        async def _run_subtitle():
            from app.services.subtitle import SubtitleEngine
            engine = SubtitleEngine(
                api_key=db.get_config("os_api_key", ""),
                user_agent=db.get_config("os_user_agent", "SubtitleHunter v13.2")
            )
            await engine.download_subtitle_for_task(
                db_manager=db,
                file_path=task_record["path"],
                tmdb_id=new_tmdb_id,
                media_type=body.media_type,
                imdb_id=new_imdb_id,
                target_path=task_record.get("target_path"),
                archive_id=body.task_id if body.is_archive else None
            )

        background_tasks.add_task(_run_subtitle)
        subtitle_result = "triggered"
```

### 前端反馈机制

`manual_rebuild` 接口立即返回 `{ "rebuilt": { "subtitle": "triggered" } }`，前端收到后：

```typescript
// 触发补录后，启动短轮询（每 3s 一次，最多 10 次 = 30s）
const pollSubtitleStatus = async (taskId: number) => {
  for (let i = 0; i < 10; i++) {
    await new Promise(r => setTimeout(r, 3000));
    const data = await api.getTasks({ search: String(taskId) });
    const task = data.tasks.find(t => t.id === taskId);
    if (task?.sub_status === 'success' || task?.sub_status === 'failed') {
      showToast(task.sub_status === 'success' ? '字幕下载成功 ✅' : '字幕下载失败 ❌');
      loadTasks();  // 刷新列表
      return;
    }
  }
  showToast('字幕搜索进行中，请稍后刷新...');
};
```

**为什么不用 WebSocket**：项目当前无 WebSocket 基础设施，短轮询实现成本最低，且字幕下载通常在 10s 内完成，轮询间隔 3s 足够及时。

## 八、三重折叠数据结构设计

```typescript
// 分组后的数据结构
interface MediaGroup {
  key: string;               // 作品名（title 或 file_name）
  media_type: 'movie' | 'tv';
  // 电影：直接存单条 task
  task?: Task;               // media_type==='movie'
  // 剧集：按季分组
  seasons?: Map<number, Task[]>;  // season_number → episodes[]
  // 聚合信息（Level 1 显示用）
  total_count: number;       // 总集数
  archived_count: number;    // 已归档数
  poster_path?: string;      // 取第一条有效海报
  tmdb_id?: number;
}
```

### useMemo 分组逻辑（伪代码）

```typescript
const groupedMedia = useMemo(() => {
  const map = new Map<string, MediaGroup>();
  for (const task of tasks) {
    // 分组 key：优先用 title，降级用 file_name，最终用 task.id
    const key = (task.title || task.file_name || String(task.id)).trim();
    if (!map.has(key)) {
      map.set(key, {
        key,
        media_type: task.media_type,
        seasons: new Map(),
        total_count: 0,
        archived_count: 0,
        poster_path: task.local_poster_path || task.poster_path,
        tmdb_id: task.tmdb_id,
      });
    }
    const g = map.get(key)!;
    g.total_count++;
    if (task.status === 'archived') g.archived_count++;
    if (!g.poster_path) g.poster_path = task.local_poster_path || task.poster_path;
    if (task.media_type === 'movie') {
      g.task = task;
    } else {
      const s = task.season ?? 1;
      if (!g.seasons!.has(s)) g.seasons!.set(s, []);
      g.seasons!.get(s)!.push(task);
    }
  }
  return Array.from(map.values());
}, [tasks]);
```

## 十一、Chinese-First Prompt 精确内容

**修改位置**：`ai_identify_media` 方法，`user_prompt` 的 ⚠️ 重要约束列表末尾

**新增第 6 条**：
```
6. 中文优先铁律（Chinese-First）：
   - 若文件名中存在任何中文字符（CJK Unicode 范围 \u4e00-\u9fff），
     query 必须且只能输出中文标题（如「飞驰人生」），
     严禁附加英文副标题、英文译名、拼音或罗马字。
   - 只有文件名完全不含中文字符时，才使用英文作为 query。
   - 理由：TMDB /search/movie?query=飞驰人生 的中文命中率远高于混合查询。
   - 反例（禁止）：query="飞驰人生 Pegasus"、query="Fei Chi Ren Sheng"
   - 正例（要求）：query="飞驰人生"
```
