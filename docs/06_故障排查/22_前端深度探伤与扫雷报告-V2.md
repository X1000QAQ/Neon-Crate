### ✅ 纠偏 2：`isMounted` 是过时反模式

**原误诊**：问题 2.3 建议使用 `isMounted` 阻断异步 setState

**真相**：
- React 16 时代的陈腐反模式
- React 18 已移除"在已卸载组件上 setState"的控制台警告
- 在绝大多数情况下不会导致真正的内存泄漏
- 引入 `isMounted` 布尔值只会增加代码异味

**结论**：✅ 删除此误诊，不使用 `isMounted`。

---

### ✅ 纠偏 3：`catch(() => ({}))` 是务实设计

**原误诊**：问题 1.2 和 1.3 要求强制转为 `unknown` 并写类型守卫

**真相**：
- `api.ts` 里的 `catch(() => ({}))` 是本项目的 Pragmatic（务实）兜底设计
- `catch (err: any)` 是合理的容错策略
- 要求强制转为 `unknown` 并写一堆 Type Guard 毫无业务增益
- 纯属脱裤子放屁

**结论**：✅ 删除此误诊，保留现有设计。

---

### ✅ 纠偏 4：盲目同步 Props 会冲掉用户输入

**原误诊**：问题 2.4 建议用 `useEffect` 监听 `task.season` 变化并重置 state

**真相**：
- 若用 `useEffect` 盲目监听 `task.season` 变化并重置 state
- 一旦后台轮询导致父组件传来的 `task` 刷新
- 用户在输入框打到一半的字就会被瞬间冲掉
- 这是更严重的 UX 灾难

**结论**：✅ 删除此误诊，不使用 `useEffect` 同步。

## 1 个真实病灶详解

### 病灶 [HIGH]：Dialog 幽灵记忆陷阱

**位置**：`frontend/components/media/RebuildDialog.tsx`

**病因**：
在 `RebuildDialog.tsx` 中，`if (!open) return null;` 只是隐藏了组件，并没有卸载它。组件的状态（`keyword`, `mediaType`, `season`, `episode` 等）被永久冻结在内存中。

**场景复现**：
1. 打开任务 A 的弹窗，搜索"庆余年"，选择季数 2，集数 5
2. 关闭弹窗（`open=false`，组件隐藏但不卸载）
3. 打开任务 B 的弹窗
4. **问题**：弹窗仍显示"庆余年"、季数 2、集数 5 的残留数据，而不是任务 B 的初始值

**根本原因**：
```typescript
// 当前代码
const [mediaType, setMediaType] = useState<string>(task.media_type || 'movie');
const [keyword, setKeyword]     = useState<string>(task.title || '');
const [season, setSeason]       = useState<number | ''>(task.season ?? 1);
const [episode, setEpisode]     = useState<number | ''>(task.episode ?? '');

if (!open) return null;  // ⚠️ 只隐藏，不卸载
```

`useState` 的初始值只在组件首次挂载时生效。当 `open` 从 `false` 变为 `true` 时，组件不会重新挂载，状态也不会重置。

**正解**：
需要一个专门的"开门重置器"。当 `open` 变为 `true` 且 `task.id` 变化时，用一个 `useEffect` 一次性将所有本地 state 重置为当前 `task` 的初始值。
