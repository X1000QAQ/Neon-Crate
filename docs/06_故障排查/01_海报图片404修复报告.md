# 海报图片 404 修复报告

> **文档编号：** NC-DEBUG-001  
> **日期：** 2026-03-15  
> **环境：** WSL2 / Ubuntu · Frontend: `localhost:3000` · Backend: `localhost:8000`  
> **状态：** ✅ 已解决

---

## 1. 故障描述 (Issue Description)

### 现象

本地开发环境下，海报墙（MediaWall）所有图片均无法显示，浏览器 Network 面板频繁出现：

```
GET http://localhost:3000/api/v1/public/image?path=%2Fhome%2F... 404 Not Found
```

前端 Next.js 开发服务器运行在 **3000 端口**，而图片代理接口挂载在 FastAPI 后端 **8000 端口**。所有图片请求错误地打向了 3000 端口，导致 Next.js 将其路由至 SPA 回退处理器并返回 404。

### 核心矛盾

> **寻址重力偏移（Address Gravity Shift）**：前端图片请求由于逻辑链路中的相对路径传递，最终"坠落"到当前页面所在的 3000 端口，而非后端物理图片代理的 8000 端口。

### 影响范围

| 组件 | 影响 |
|------|---------|
| `MediaTable.tsx` | 所有海报图片不可见，显示占位灰块 |
| `SecureImage.tsx` | fetch 请求目标错误，Blob URL 无法生成 |
| 用户体验 | 海报墙全黑，无法识别媒体内容 |

---

## 2. 根本原因分析 (Root Cause Analysis)

### 2.1 逻辑闭锁：`getPosterUrl` 提前拼接相对路径

**这是核心 Bug。**

`MediaTable.tsx` 中的 `getPosterUrl` 函数在将路径传入 `SecureImage` 之前，已经将原始物理路径（`/home/x1000qaq/.../poster.jpg`）拼接成了相对 URL：

```typescript
// ❌ 修复前：getPosterUrl 提前拼接
const getPosterUrl = (task: Task): string => {
  const posterPath = task.local_poster_path || task.poster_path;
  if (!posterPath) return '/placeholder-poster.jpg';
  if (posterPath.startsWith('http://') || posterPath.startsWith('https://')) {
    return posterPath;
  }
  // 问题根源：直接返回相对路径
  return `/api/v1/public/image?path=${encodeURIComponent(posterPath)}`;
};
```

`SecureImage.tsx` 收到 `/api/v1/public/image?path=...` 后，其防套娃逻辑命中：

```typescript
// SecureImage 防套娃检测
const targetPath = src.includes('/public/image?path=')
  ? src           // ← 命中！直接使用 src，即相对路径
  : `${API_BASE}/public/image?path=${encodeURIComponent(src)}`;
```

由于相对路径命中了防套娃条件，`SecureImage` 直接将相对路径设为 `finalUrl`，最终 `fetch` 请求打向 `localhost:3000`。

**完整链路崩溃图：**

```
DB: /home/x1000qaq/.../poster.jpg
        ↓
getPosterUrl()  →  /api/v1/public/image?path=%2Fhome%2F...  ← 相对路径（BUG）
        ↓
SecureImage 接收 src
        ↓
src.includes('/public/image?path=') → true → 防套娃命中
        ↓
setFinalUrl(相对路径)  →  setBlobUrl(相对路径)
        ↓
fetch('/api/v1/public/image?...')  →  打向 localhost:3000  →  404 ❌
```

### 2.2 权限与缓存冲突：root 进程锁死 `.next` 缓存

由于前端曾以 `sudo npm run dev` 启动，Next.js 生成的 `.next` 编译缓存文件属主变为 `root`：

```bash
-rwxr-xr-x 1 root root 4208 Mar 15 12:55 SecureImage.tsx
```

即使通过 Cursor IDE 写入了修复代码，热重载（HMR）读取的仍是 root 锁死的旧版文件，导致代码修改「无声无效」，极难排查。

### 2.3 配置契约缺失：`APP_ENV` 未在 Pydantic Settings 中定义

`system.py` 的白名单扩容逻辑依赖 `settings.APP_ENV` 判断开发环境：

```python
dev_mode = _settings.APP_ENV.lower() in ("development", "dev", "local")
```

但 `app/infra/config/__init__.py` 的 `Settings` 模型中未声明 `APP_ENV` 字段，导致运行时 AttributeError，白名单扩容失败，WSL 本地路径（`/home/x1000qaq/...`）被拒绝访问，返回 403。

## 3. 修复方案 (Resolution)

### 3.1 前端原子级重构

#### `MediaTable.tsx` — 移除提前拼接

```typescript
// ✅ 修复后：直接返回原始物理路径，由 SecureImage 统一处理
const getPosterUrl = (task: Task): string => {
  const posterPath = task.local_poster_path || task.poster_path;
  if (!posterPath) return '/placeholder-poster.jpg';
  // 直接返回原始路径，不在此处拼接
  // 避免相对路径绕过 SecureImage 的绝对地址锁定逻辑
  return posterPath;
};
```

#### `SecureImage.tsx` — 物理硬连后端地址

采用两种方案迭代，最终落地方案为直接信任 `config.ts` 中的 `API_BASE`：

```typescript
// ✅ 最终方案：使用全局 API_BASE（编译期由 .env.local 注入绝对路径）
import { API_BASE } from '@/lib/config';

useEffect(() => {
  if (!src) return;

  // 外部图片（TMDB 等）直通
  if (src.startsWith('http')) {
    setFinalUrl(src);
    return;
  }

  // 使用全局 API_BASE（本地 = http://localhost:8000/api/v1）
  const targetPath = src.includes('/public/image?path=')
    ? src
    : `${API_BASE}/public/image?path=${encodeURIComponent(src)}`;

  setFinalUrl(targetPath);
  console.warn('🚀 [FINAL_FIX] 物理地址锁定:', targetPath);
}, [src]);
```

同时加入 HTTP 响应状态检查，防止后端返回 403 HTML 被当成图片渲染：

```typescript
const res = await fetch(finalUrl, {
  headers: { Authorization: `Bearer ${token ?? ''}` },
});

// ✅ 关键防御：res.ok 检查，避免将错误页当成 Blob 渲染
if (!res.ok) throw new Error(`HTTP ${res.status}`);

const blob = await res.blob();
```

### 3.2 后端防线优化

#### `config/__init__.py` — 纳入 `APP_ENV` 配置契约

```python
class Settings(BaseSettings):
    # ... 其他字段
    APP_ENV: str = "production"  # development | production
```

对应 `.env` 文件：

```bash
APP_ENV=development
```

#### `system.py` — 白名单扩容 + 成功日志

```python
# 开发模式：追加 home 目录至白名单
dev_mode = _settings.APP_ENV.lower() in ("development", "dev", "local")
if dev_mode:
    home_path = Path.home()

# 图片代理成功诊断日志
logger.info(f"[IMAGE_ACCESS_SUCCESS] ✅ 图片代理成功: {normalized_path}")
return FileResponse(str(img_path))
```

## 4. 验证与结论 (Verification)

### 验证链路

| 验证点 | 预期值 | 结果 |
|--------|--------|------|
| 浏览器 Console | `🚀 [FINAL_FIX] 物理地址锁定: http://localhost:8000/api/v1/...` | ✅ |
| Network 请求目标 | `http://localhost:8000/api/v1/public/image?path=...` | ✅ |
| HTTP 状态码 | `200 OK` | ✅ |
| 后端日志 | `[IMAGE_ACCESS_SUCCESS] ✅ 图片代理成功: /home/x1000qaq/...` | ✅ |
| 海报墙显示 | 全量图片正常渲染 | ✅ |

### 关键教训

1. **单一职责原则**：`getPosterUrl` 只应负责「选取哪个路径」，不应负责「如何构建 URL」，后者应由 `SecureImage` 统一处理。

2. **防套娃逻辑是双刃剑**：防套娃条件（`includes('/public/image?path=')`）是正确的，但上游如果已经拼接了相对路径传入，会导致误判。根治方案是规范上游的输入格式。

3. **永不用 `sudo` 运行开发服务器**：root 进程会将缓存文件锁死为 root 属主，导致后续修改静默失效，是排查中最危险的「无声 Bug」。

4. **配置契约完整性**：所有在运行时通过 `settings.xxx` 访问的字段，必须在 Pydantic `Settings` 模型中显式声明，否则运行时 AttributeError 会静默降级，掩盖真正的问题。

## 6. AIO 生产环境追加分析（2026-03-15）

### 问题 B1：`NEXT_PUBLIC_API_BASE` 编译期固化，局域网设备海报失效

`NEXT_PUBLIC_*` 变量在 `next build` 静态导出时被 bake 进 JS bundle，运行时不可变。原 `frontend/.env.local` 中配置 `http://localhost:8000/api/v1` 会随 Docker 镜像固化：

```
用户从手机/平板访问 http://192.168.1.x:8000
    ↓
SecureImage 请求 http://localhost:8000/...（bundle 中写死）
    ↓ localhost 指客户端自身，不是 NAS
    ❌ 连接失败，海报全黑
```

**修复**：`NEXT_PUBLIC_API_BASE` 迁至 `.env.development.local`，AIO 构建不设置该变量，`API_BASE` 降级为相对路径 `/api/v1`。

### 问题 B2：AIO 相对路径被 `<img>` 直接渲染导致 401

原逻辑 `!finalUrl.startsWith('http')` → 直接 `setBlobUrl`，用 `<img src>` 渲染。但 AIO 模式下 `finalUrl = '/api/v1/public/image?path=...'`，`<img>` 标签不携带 `Authorization`，而 `public_router` 有 JWT 依赖 → **401**。

**修复**：只有「绝对 URL 且不含 `/public/image?path=`」的外部图片才直接渲染，其余一律走 `fetch + token`。
