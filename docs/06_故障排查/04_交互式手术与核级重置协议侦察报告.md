## 二、Bug 1 分析：UI 越界 — 电影被错误折叠

### 现象

`MediaTable.tsx` 当前的 `useMemo` 分组逻辑：

```typescript
// 当前代码（有 Bug）
const key = (task.title || task.file_name || String(task.id)).trim();
if (task.media_type === 'movie') {
  g.task = task;  // 电影存入 group.task
} else {
  const s = task.season ?? 1;
  g.seasons.get(s)!.push(task);  // 剧集存入 seasons Map
}
```

渲染层：

```typescript
// 当前渲染（有 Bug）
{group.media_type === 'movie' && group.task && (
  // 电影：直接渲染 TaskRow ← 正确
  <TaskRow task={group.task} ... />
)}
```

### 根因

**电影渲染逻辑本身正确**（`group.media_type === 'movie'` 直接渲染 `TaskRow`，不走 Season→Episode 折叠）。

但存在一个**分组键冲突问题**：同名电影的多个任务（如重复扫描）会被 `key = task.title` 归入同一个 group，导致后者覆盖前者（`g.task = task` 只保留最后一条），前几条电影任务从视图中消失。

**真正的越界 Bug** 是：若某电影任务的 `task.title` 恰好与某剧集同名（极端情况），或 `task.media_type` 字段为 `null`/`undefined` 时，该任务会被归入错误的 group 分支，进入 `seasons` Map 渲染为剧集折叠节点。

### 修复方案

**分组键加入 `media_type` 作为命名空间隔离**，防止电影与剧集同名冲突：

```typescript
// 修复后
const key = `${task.media_type || 'movie'}::${(task.title || task.file_name || String(task.id)).trim()}`;
```

**每个 group 的 `media_type` 由第一条任务决定，后续任务严格按 `media_type` 路由**：

```typescript
if ((task.media_type || 'movie') === 'movie') {
  g.task = task;
} else {
  const s = task.season ?? 1;
  if (!g.seasons.has(s)) g.seasons.set(s, []);
  g.seasons.get(s)!.push(task);
}
```

**Level 1 节点：电影不渲染折叠箭头，直接展示 TaskRow**（不需要 `ChevronDown`/`ChevronRight`，不需要 `openL1` 状态控制）：

```tsx
{/* 电影：Level 1 直接平铺，无折叠 */}
{group.media_type === 'movie' && group.task && (
  <TaskRow task={group.task} ... />
)}

{/* 剧集：Level 1 折叠按钮 + Level 2/3 嵌套 */}
{group.media_type === 'tv' && (
  <>
    <button onClick={() => toggleL1(group.key)}>...</button>
    <div style={{ maxHeight: l1Open ? '9999px' : '0px' }}>...</div>
  </>
)}
```

## 三、Bug 2 分析：数据幽灵 — 数据库未同步写入

### 现象核查

读取 `task_repo.py` 中 `update_any_task_metadata` 的完整实现（第 396–440 行），发现以下关键问题：

#### 问题 A：`local_poster_path` 参数不支持

`db_manager.py` 的代理方法签名：

```python
def update_any_task_metadata(self, task_id, is_archive, imdb_id=None, tmdb_id=None,
                              sub_status=None, title=None, year=None):
```

`manual_rebuild` 中调用：

```python
# 当前代码（有 Bug）— local_poster_path 参数不被接受！
db.update_any_task_metadata(body.task_id, body.is_archive, local_poster_path=local_poster)
```

`db_manager.py` 的代理方法**没有 `local_poster_path` 参数**，`task_repo.py` 的底层实现同样没有该列的 UPDATE 支持。这个调用**会被 Python 以 `TypeError: unexpected keyword argument` 静默吞掉**（因为外层有宽泛的 `except Exception`），导致海报路径**永远不会写入数据库**。

#### 问题 B：`is_archive=True` 时使用 `original_task_id` 匹配

`task_repo.py` 中：

```python
if is_archive:
    table = "media_archive"
    pk_field = "original_task_id"  # ← 冷表主键是 original_task_id，不是 id！
else:
    table = "tasks"
    pk_field = "id"
```

`manual_rebuild` 中传入的是 `body.task_id`，这是前端传来的 ID。问题在于：**前端的 `task_id` 到底是 `media_archive.id` 还是 `media_archive.original_task_id`？**

读取 `get_archived_data()` 返回的数据结构（需进一步确认），若返回的是 `media_archive.id`（冷表自增主键），而底层用 `original_task_id` 匹配，则会导致**零更新（rowcount=0）**，即"数据幽灵"现象。

#### 问题 C：两次调用 `update_any_task_metadata`

```python
# 第一次调用
db.update_any_task_metadata(body.task_id, body.is_archive,
    tmdb_id=new_tmdb_id, imdb_id=..., title=..., year=...,
    sub_status="pending" if body.refix_subtitle else None)

# 第二次调用（有 Bug：参数名不存在）
if local_poster:
    db.update_any_task_metadata(body.task_id, body.is_archive, local_poster_path=local_poster)
```

将两次调用合并为一次，并修复 `local_poster_path` 支持。

### 修复方案

**Step 1**：在 `task_repo.py` 的 `update_any_task_metadata` 中新增 `local_poster_path` 参数支持：


## 五、核级重置协议分析

### 5.1 物理目录"核级清理"

现有 `_safe_delete_metadata_files` 只删白名单文件（精准清理），不满足"核级清理"需求。
需新增 `_nuclear_clean_directory`：

```python
VIDEO_EXTENSIONS = {'.mkv', '.mp4', '.avi', '.ts', '.m2ts', '.mov', '.wmv', '.flv', '.rmvb', '.webm'}

def _nuclear_clean_directory(metadata_dir: str, library_root: str) -> dict:
    """核级清理：删除 metadata_dir 下除视频本体以外的所有文件。双重路径防穿越。"""
    resolved_meta = Path(metadata_dir).resolve()
    resolved_lib  = Path(library_root).resolve()
    try:
        resolved_meta.relative_to(resolved_lib)
    except ValueError:
        raise PermissionError(f"[NUCLEAR] 路径越权: {metadata_dir}")
    deleted, kept = [], []
    for f in resolved_meta.iterdir():
        if not f.is_file(): continue
        try: f.resolve().relative_to(resolved_meta)
        except ValueError: continue
        if f.suffix.lower() in VIDEO_EXTENSIONS:
            kept.append(str(f))
        else:
            f.unlink(); deleted.append(str(f))
    return {"deleted": deleted, "kept_videos": kept}
```

### 5.2 视频文件重命名

```python
def _rename_video_file(video_path, title, year, media_type, season=None, episode=None) -> str:
    safe_title = re.sub(r'[\\/:*?"|<>]', '_', title.strip())
    year_str = f" ({year})" if year else ""
    ep_str = f" - S{str(season).zfill(2)}E{str(episode).zfill(2)}" \
             if media_type == 'tv' and season is not None and episode is not None else ""
    new_name = f"{safe_title}{year_str}{ep_str}{Path(video_path).suffix}"
    new_path = str(Path(video_path).parent / new_name)
    if video_path != new_path:
        os.rename(video_path, new_path)
    return new_path
```

### 5.3 数据库路径同步（Critical）

重命名后必须立即同步 `target_path`，否则 404。
`update_any_task_metadata` 当前**不支持 `target_path`**，需新增该参数。

**执行顺序（必须严格遵守）**：
```
1. 获取 TMDB 元数据
2. _nuclear_clean_directory()   ← 清理非视频文件
3. _rename_video_file()         ← 物理重命名，得到 new_video_path
4. db.update_any_task_metadata(..., target_path=new_video_path)  ← 立即同步！
5. generate_nfo()  →  download_poster()  →  refix_subtitle
```

步骤 3→4 必须绑定，重命名成功但 DB 失败时回滚文件名：
```python
try:
    new_video_path = _rename_video_file(old_path, ...)
    db.update_any_task_metadata(..., target_path=new_video_path)
except Exception as e:
    if new_video_path != old_path and Path(new_video_path).exists():
        os.rename(new_video_path, old_path)  # 回滚
    raise HTTPException(status_code=500, detail=str(e))
```

## 六、关键发现：ID 匹配错误（数据幽灵根因 — 已确认）

### 根因

`get_archived_data()` 返回 `media_archive.id`（自增 PK）作为 `"id"` 字段。

`update_any_task_metadata(is_archive=True)` 用 `original_task_id` 匹配：

```python
pk_field = "original_task_id"  # ← 不是 id！
```

**`media_archive.id` ≠ `media_archive.original_task_id`**，导致 `WHERE original_task_id = {archive.id}` 零命中，所有元数据更新被静默丢弃。

### 修复方案（推荐方案 B：最小改动）

修改 `task_repo.py`，`is_archive=True` 时改用 `id` 匹配冷表：

```python
if is_archive:
    table = "media_archive"
    pk_field = "id"  # ← 修复：用冷表自增 PK 匹配，与 get_archived_data 返回值一致
else:
    table = "tasks"
    pk_field = "id"
```

> ⚠️ **CRITICAL 影响评估**：此修改影响 `perform_find_subtitles_task_sync` 和 `perform_scrape_all_task_sync` 两个调用方。需确认这两个调用方传入的 `task_id` 是否也是 `media_archive.id`（而非 `original_task_id`）。

调用方检查结论（来自 GitNexus 图谱）：
- `perform_find_subtitles_task_sync` 传入 `archive_id`（来自 `get_archived_data()["id"]`）→ **是 `media_archive.id`，改方案 B 正确**
- `perform_scrape_all_task_sync` 传入 `task_id`（来自 `get_tasks_needing_scrape()`，操作 `tasks` 表，`is_archive=False`）→ **不受影响**
