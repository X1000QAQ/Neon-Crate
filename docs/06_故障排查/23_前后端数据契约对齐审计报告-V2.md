## 机长纠偏确认

### ✅ 纠偏 1：`year` 和 `tmdb_id` 类型自动转换

**原误诊**：问题 M1 认为 `year` 和 `tmdb_id` 类型不匹配

**真相**：
- SQLite 建表语句是 TEXT，但 `domain_media.py` 中定义的是 `year: Optional[int] = None`
- FastAPI/Pydantic 会在序列化时自动将数据库的字符串强转为 JSON Number
- 前端用 `number` 接收是 100% 正确的

**结论**：✅ 删除此误诊，类型转换由 Pydantic 自动处理。

---

### ✅ 纠偏 2：冷表 ID 已在 SQL 层伪装

**原误诊**：问题 2.2a 认为冷表 ID 映射不清晰

**真相**：
在 `task_repo.py` 的冷表查询中，已经明确写了：
```python
SELECT original_task_id AS id, ...
```

我们在 SQL 层面已经完成了伪装，前端拿到的永远是统一的 `id`，不需要区分热表主键还是冷表主键。

**结论**：✅ 删除此误诊，冷热表已透明化。

---

### ✅ 纠偏 3：严格枚举是刻意设计

**原误诊**：建议 1 建议将严格枚举降级为 string

**真相**：
- 前端的 `media_type: 'movie' | 'tv'` 和 `status` 等联合类型是刻意设计的强类型契约
- 将其降级为 `string` 属于开历史倒车的弱智行为

**结论**：✅ 删除此误诊，保留严格枚举。

---

### ✅ 纠偏 4：`last_sub_check` 是垃圾字段

**原误诊**：问题 H1 建议把 `last_sub_check` 加到前端

**真相**：
- 前端 UI 根本不需要展示"上次字幕检查时间"
- 后端查出来了不代表前端接口就必须接住
- 不要给前端模型塞垃圾字段

**结论**：✅ 删除此误诊，不添加 `last_sub_check`。

## 1 个真实病灶详解

### 病灶 [HIGH]：`is_archive` 跨端隐身术

**病因**：
后端 `task_repo.py` 的查询返回了 `is_archive` 标志位（第 218、243 行明确写了 `0 as is_archive` 和 `1 as is_archive`），但在 `domain_media.py` 的 `Task` Pydantic 模型中并未定义该字段。导致 FastAPI 响应时将其当做未定义数据直接丢弃！

**后果**：
- 前端 `index.ts` 的 `Task` 接口也缺少此字段
- 导致弹窗在触发 `rebuildTask` 时，永远丢失冷热表坐标
- 造成更新打空（`update_any_task_metadata` 无法正确判断应该更新热表还是冷表）

**场景复现**：
1. 用户打开一个已归档任务的弹窗（来自冷表，`is_archive=True`）
2. 执行 `rebuildTask`，前端无法传递 `is_archive` 标志
3. 后端 `update_any_task_metadata` 收不到 `is_archive`，默认更新热表
4. 冷表数据未被更新，导致数据不一致

## 修复后的数据流

```

  ↓

  └─ is_archive?: boolean
  ↓

  ↓
后端 update_any_task_metadata
  └─ 根据 is_archive 选择热表或冷表更新
```
