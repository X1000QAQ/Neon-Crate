# 文件归档与存储防御

---

## 一句话理解

把下载目录里的文件「无损搬运」进媒体库，同时保证数据库在高并发下不出错，并防止字幕 API 被刷爆。

---

## 硬链接归档：零空间占用

刮削完成后，系统不会「复制」文件，而是创建**硬链接**：

```
下载目录/电影.mkv  ──(硬链接)──→  媒体库/电影/电影 (2008)/电影 (2008).mkv
```

- 磁盘上只有一份数据，但两个路径都能访问
- 删除下载目录的原文件，媒体库里的不受影响
- 零额外空间占用

**跨盘情况：** 如果下载盘和媒体库不在同一个分区，硬链接不支持，自动降级为**软链接（符号链接）**。软链接不占空间，但原文件删了媒体库链接就失效了，建议下载盘和媒体库放同一分区。

---

## 字幕伴生迁移

如果下载目录里有字幕文件，归档时自动一起迁移：

```
下载目录/
  电影.mkv
  电影.zh-CN.srt   ← 自动识别并迁移

归档后媒体库/
  电影 (2008).mkv
  电影 (2008).zh-CN.srt   ← 命名自动对齐
```

**安全规则：** 只迁移「同目录、同文件名前缀」的字幕，不会误把别的电影的字幕搬过来。如果目录里放了很多不同电影（下载大厅目录），自动跳过不处理。

---

## 媒体库 1+1 约束

系统强制要求路径配置满足：
- **电影媒体库：恰好 1 个启用**
- **剧集媒体库：恰好 1 个启用**

超过 1 个会报错。这保证所有归档都有唯一的目标位置，不会出现「同一部电影散落在多个目录」的混乱。

---

## 双表去重防重复归档

`tasks` 是活跃任务表，`media_archive` 是已归档记录表。扫描入库前，`check_task_exists_by_path()` 同时检查两张表：

```
SELECT 1 FROM tasks WHERE path = ?           ← 检查活跃任务
SELECT 1 FROM media_archive WHERE path = ?   ← 检查归档记录
```

任何一张表命中 → 跳过入库，防止已归档文件被重复处理。

**删除任务时的注意事项：** 前端删除任务会同时清理两张表的记录（按 `path` 匹配），否则 `media_archive` 残留会导致文件永远无法重新入库。旧版按 `id` 删除存在 bug（两表 id 独立自增，不对应），已修复。

---

## 数据库高并发保护

扫描、刮削、字幕、下载这些任务都在同时跑，都要读写数据库。系统通过两层机制防止冲突：

**WAL 模式（Write-Ahead Logging）：**
- 写入时先记日志，多个读操作不会被写锁阻塞
- 显著减少「database is locked」错误

**全局互斥锁：**
- 所有写操作排队执行，避免多个线程同时写入造成数据损坏
- 读操作也在锁保护下执行，确保连接不被跨线程误关闭

---

## 字幕 API 冷却机制（防刷爆）

冷门片子在字幕库里可能永远没有中文字幕，如果每次任务都重新搜索，会白白消耗 OpenSubtitles 的 API 配额。

**7天冷却规则：**

| 字幕状态 | 下次搜索时机 |
|---|---|
| `pending`（从未搜索过）| 立即搜索 |
| `failed`（搜索出错）| 立即重试 |
| `scraped`（已找到）| 不再搜索 |
| `missing`（搜到但没有中文字幕）| 7天后再试 |

这样冷门片子不会被每天重复轮询，API 配额留给真正需要的任务。
