### 2.1 启动序列

```
应用启动
  ├─► 日志系统初始化（RotatingFileHandler，10MB × 5）
  ├─► DatabaseManager 懒初始化
  │     ├─► SQLite WAL 模式

  │     ├─► _migrate_sensitive_keys()   历史明文密钥迁移
  │     └─► _inject_ai_defaults()       创世配置自愈注入（Genesis Config Healing）
  │           **绕过 get_config() 的 DEFAULT_CONFIG/内存兜底层**，直接读取磁盘 `config.json` 原始值，
  │           对 7 个核心字段执行“缺啥补啥”的**物理注入**（AI 人格/正则/文件格式），
  │           绝不覆盖用户已填写的非空值，幂等安全
  ├─► 路由注册（auth / public / api_router）
  ├─► 静态资源挂载（/api/v1/assets → /storage）
  ├─► AIO 前端挂载（static/ → /，html=True）
  ├─► SPA 404 回退处理器注册
  └─► asyncio.create_task(cron_scanner_loop())  后台巡逻启动
```

### 2.2 自动巡逻流水线（cron_scanner_loop）

```
每轮：读取 cron_enabled
  = False → 跳过
  = True  → 1. 物理扫描
          → 读取 auto_scrape = True → 2. 全量刮削
          → 读取 auto_subtitles = True → 3. 字幕查找
等待 cron_interval_min 分钟后下一轮
```

| 配置键 | 默认值 | 说明 |
|--------|--------|------|
| `cron_enabled` | False | 巡逻总开关 |
| `cron_interval_min` | 60 | 间隔（分钟，≤0 回退到 60）|
| `auto_scrape` | False | 扫描后自动刮削 |
| `auto_subtitles` | False | 刮削后自动找字幕 |

### 2.3 路由优先级

```
1. /api/v1/auth/*    鉴权路由（显式豁免 JWT）
2. /api/v1/public/*  公共路由（JWT 保护）
```

### 3.2 数据库双表设计（v1.0.0）

#### 热表 vs 冷表

| 特性 | tasks（热表）| media_archive（冷表）|
|------|-------------|-------------------|
| 用途 | 新任务、处理中、已处理 | 已归档、历史记录 |
| 主键 | id | original_task_id |
| 状态 | pending/archived/failed/ignored | archived（固定）|
| 查询频率 | 高（前端列表、统计）| 低（历史查询）|
| 更新频率 | 高（状态变化）| 低（元数据补全）|
| 前端显示 | 可编辑、可删除 | 只读、可重建 |

#### 双表联查（UNION ALL）

```python
# stats_repo.py: get_all_data()
# 业务链路：1. 构建双表 UNION ALL 查询 -> 
#          2. 统一使用 original_task_id 作为冷表的对外 ID -> 
#          3. 应用搜索过滤（可选）-> 
#          4. 按创建时间倒序排列

base_query = f"""
    SELECT id, path, file_name, clean_name, type, status, tmdb_id, imdb_id, title, year, 
           target_path, sub_status, last_sub_check, created_at, poster_path, local_poster_path, 
           season, episode, 0 as is_archive
    FROM tasks 
    WHERE 1=1 {ignored_clause}
    UNION ALL
    SELECT original_task_id as id, path, file_name, clean_name, type, 'archived' as status, 
           tmdb_id, imdb_id, title, year, target_path, sub_status, NULL as last_sub_check, 
           created_at, poster_path, local_poster_path, season, episode, 1 as is_archive
    FROM media_archive
    WHERE 1=1
    ORDER BY created_at DESC
"""
```

#### is_archive 跨端传递

```python

  "is_archive": true,  # 🚨 必须传递：0=热表，1=冷表

# 后端处理
if is_archive:
```

### 3.3 数据库迁移引擎（v1.0.0）

#### 版本迁移机制（_migrate_database）

**业务链路**：
```
1. 读取 system_meta 中的当前 schema_version -> 
2. 遍历 MIGRATIONS，执行所有版本号 > 当前版本的任务 -> 
3. 每个迁移任务独占一个事务（BEGIN ... COMMIT / ROLLBACK）-> 
4. 迁移完成后更新 schema_version -> 
5. 任何迁移失败立即 rollback 并抛出异常，阻止系统启动
```

#### 敏感密钥迁移（_migrate_sensitive_keys）

**业务链路**：
```
1. 读取 config.json 文件 -> 
2. 扫描 SENSITIVE_KEYS 列表中的明文密钥 -> 
3. 将非空明文密钥加密后写入 secure_keys.json -> 
4. 清空 config.json 中的明文密钥 -> 
5. 后续读取时，ConfigRepo 会自动从 secure_keys.json 解密
```

**幂等性**：多次执行不会重复迁移（已加密的密钥会被跳过）。
| 线程安全 | `threading.local()` 线程级连接池 |
| 敏感密钥加密 | 6个 API Key 自动加解密 |
| AI 规则注入 | 首次启动注入人格/路由规则/归档规则/15条正则 |

### 3.2 安全模块（security/crypto.py）

**神盾计划（Project Aegis）** — 四项安全能力：

| 能力 | 实现 | 用途 |
|------|------|------|
| Fernet 对称加密 | `cryptography.fernet` | 加密存储 API 密钥 |
| Bcrypt 密码哈希 | `passlib[bcrypt]` | 管理员账号密码 |
| JWT Token | `PyJWT` HS256 | 无状态会话（7天）|
| 管理员账号管理 | `data/auth.json` | 初始化/验证/读取 |

```python
crypto = get_crypto_manager()
crypto.encrypt_api_key("sk-xxx")     # → 密文
crypto.decrypt_api_key(cipher)       # → 明文，失败返回空字符串
crypto.create_access_token("admin")  # → JWT（7天）
crypto.verify_token(token)           # → username 或 None
crypto.init_admin(username, pwd)     # 仅允许执行一次
```

### 4.4 并发防重机制（v1.0.0-Stable 核心加固）

每个任务模块独立维护 `threading.Lock` + 状态字典：

```python
# 物理级并发防重逻辑：
# 1. 非阻塞获取锁（blocking=False），失败立即返回丢弃冗余请求
# -> 2. 检查 is_running 内存标记（双重防护）
# -> 3. finally 块无条件释放锁（系统自愈，防按钮永久僵死）
if not _scan_entry_lock.acquire(blocking=False):
    logger.warning("[SCAN] 拦截并发请求，本次触发已丢弃")
    return
```

### 5.1 AI 决策层（services/ai/）

**AIAgent** — 所有配置从数据库动态读取，WebUI 修改无需重启。

意图常量：

| 常量 | 含义 |
|------|------|
| `ACTION_SCAN` | 物理扫描 |
| `ACTION_SCRAPE` | 全量刮削 |
| `ACTION_SUBTITLE` | 字幕查找 |
| `DOWNLOAD` | 下载指令 |
| `CHAT` | 普通聊天 |

`process_message()` 流程：
```
1. 候选等待状态拦截（读取 DB _pending_candidates）
-> 2. LLMClient.call_llm() 意图识别（JSON 输出）
-> 3. 解析失败 → 关键词规则引擎兜底
-> 4. _generate_llm_response() 生成响应
     ├─► DOWNLOAD（模糊）→ TMDB 候选 → 写入 _pending_candidates
     ├─► DOWNLOAD（精准）→ ServarrClient 直接下载
     └─► 其他 → LLM 生成回复
```

**LLMClient** — 兼容所有 OpenAI 格式 API，三连击指数退避重试（1s→2s→4s）。

### 6.1 流水线编排器模式（Pipeline Orchestrator Pattern）

> **[v1.0.0 新增]** 全量刮削任务已完成「上帝函数原子化剥离」重构。原 600+ 行单体函数被解构为 1 个纯粹编排器 + 4 个职责单一的原子步骤函数。

#### 设计动机

旧架构中，`_process_single_task` 是一个承载全部业务逻辑的上帝函数（God Function），导致：
- 单步失败难以定位
- 无法对单个步骤做独立容错
- 测试覆盖几乎不可能
- 状态耦合导致难以维护

#### 编排器架构

```
_process_single_task()  ← 纯粹编排器，不含业务逻辑
  │
  ├─► _step_nfo_shortcut()           # Step 1: NFO 短路拦截
  ├─► _step_ai_extraction()          # Step 2: AI 提炼
  ├─► _step_tmdb_search_and_dup_check()  # Step 3: TMDB 搜索与防重
  └─► _step_archive_and_metadata()   # Step 4: 就地补录/归档全链路
```

#### `(is_success, is_failed)` 信号机制

编排器通过返回 `(bool, bool)` 元组实现主循环与具体业务步骤的**完全解耦**：

```python
# 主循环（perform_scrape_all_task_sync）
for task in tasks_to_scrape:
    is_success, is_failed = _process_single_task(..., task=task)
    if is_success:
        success_count += 1
    if is_failed:
        failed_count += 1
    processed += 1
```

```python
# 编排器（_process_single_task）
# 原子步骤通过返回值驱动流程，不抛出异常
if _step_nfo_shortcut(...):
    return True, False   # 短路成功

tmdb_data = _step_tmdb_search_and_dup_check(...)
if not tmdb_data:
    return False, True   # TMDB 失败或防重命中

target_path = _step_archive_and_metadata(...)
if target_path is None and not task.get("status") == "archived":
    return False, True   # 归档失败

return True, False       # 全链路成功
```

**信号语义表**：

| `(is_success, is_failed)` | 含义 |
|---------------------------|------|
| `(True, False)` | 全链路成功（NFO 短路 / 正常归档 / 存量省流）|
| `(False, True)` | 处理失败（TMDB 无结果 / 防重命中 / 归档异常）|
| `(False, False)` | 保底返回（理论上不可达，防御性编程）|

#### 原子步骤独立容错原则

每个 `_step_xxx` 函数：
1. **自包含**：所有依赖通过参数传入，无隐式全局状态
2. **独立容错**：内部异常自行捕获并降级，不上抛崩溃主流程
3. **职责单一**：只做一件事，边界清晰
4. **返回值语义明确**：`bool`（短路判断）或 `dict|None`（数据传递）

### 6.2 并发安全加固（Issue 4.1/4.2 物理修复）

#### Issue 4.1 修复：锁内二次状态复检

**问题**：旧代码在拿到 `threading.Lock` **之前**就检查 `is_running`，存在极端竞态窗口——两个线程可能同时通过状态检查，但只有一个能拿到锁，另一个在拿到锁后未再次检查就继续执行。

**修复方案**：在物理锁保护下执行二次状态复检，彻底消除竞态窗口：

```python
# 第一道防线：物理锁（拒绝排队等待，直接丢弃）
if not _scrape_entry_lock.acquire(blocking=False):
    logger.warning("[SCRAPE] 拦截并发请求，本次触发已丢弃")
    return

# 第二道防线：锁内二次状态复检（消除竞态窗口）
# 在锁保护下再次验证状态，确保任何情况下都不会双重执行
if scrape_all_status["is_running"]:
    _scrape_entry_lock.release()  # 主动释放，避免死锁
    logger.warning("[SCRAPE] 锁内二次检查拦截：状态仍为 running，释放锁并退出")
    return

try:
    scrape_all_status.set_running(True, clear_error=True)  # 原子操作
    ...
finally:
    scrape_all_status.set_running(False)  # 无条件复位
    _scrape_entry_lock.release()           # 无条件释放（系统自愈）
```

#### Issue 4.2 修复：ScrapeStatusManager 原子化

**问题**：`scrape_all_status` 为裸字典，多线程并发读写无保护，存在数据竞态（TOCTOU）。

**修复方案**：升级为 `ScrapeStatusManager` 类，内部 `threading.Lock` 保护所有状态操作：

```python
class ScrapeStatusManager:
    def __init__(self):
        self._lock = threading.Lock()
        self._state = {"is_running": False, "last_run_time": None,
                       "processed_count": 0, "error": None}

    def set_running(self, is_running: bool, clear_error: bool = False):
        with self._lock:   # 原子操作
            self._state["is_running"] = is_running
            if clear_error:
                self._state["error"] = None

    def finish_run(self, processed_count: int):
        import time
        with self._lock:   # 计数 + 时间戳原子写入，不可分割
            self._state["processed_count"] = processed_count
            self._state["last_run_time"] = time.time()

    def set_error(self, error: str):
        with self._lock:
            self._state["error"] = error

    def get_snapshot(self) -> dict:
        with self._lock:   # 深拷贝，快照与状态解耦
            return dict(self._state)

# _shared.py 中实例化
scrape_all_status = ScrapeStatusManager()
```

**向后兼容**：`__getitem__`/`__setitem__`/`get()` 接口保持字典语法可用，路由状态查询接口无需修改。
