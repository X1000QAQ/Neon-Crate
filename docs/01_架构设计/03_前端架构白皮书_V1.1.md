## 2. 系统定位与设计哲学

### 2.1 系统定位

Neon Crate 前端是**「数据容器编排引擎的工业级霓虹控制面板」**，承担四大职责：

- **可视化总控台**：媒体墙任务列表、统计面板、实时日志、系统监控核心仪表盘
- **AI 交互前端舱**：对接后端 `agent.py` 总控神经，提供自然语言指令与候选下载 UI
- **配置与策略中台**：通过 `SettingsHub` 及其子面板承载路径、下载器、LLM、AI 人设、正则规则等配置
- **错误与异常终端**：在极端异常场景下提供稳定、明确、可恢复的 UI 降级界面

### 2.2 设计哲学

| 原则 | 具体体现 |
|------|----------|
| **类型即契约** | `types/index.ts` 是前端唯一类型来源，与后端 Pydantic 模型严格对齐 |
| **单一数据入口** | 所有 HTTP 调用只能经过 `lib/api.ts`，组件禁止裸调 `fetch` |
| **组件职责单一** | 容器组件持有状态、展示组件接受 props，禁止巨石组件形态 |
| **国际化优先** | 所有 UI 字符串通过 `t(key)` 获取，严禁硬编码中英文字面量 |
| **错误防线纵深** | `apiError.ts` → `NeuralLinkAlert` → `error.tsx` 三层递进容错 |
| **主题一致性** | 所有组件遵循霓虹主题规范，统一使用 CSS 变量，禁止离散色值 |
| **前端不存业务规则** | 正则规则、AI 规则来自后端数据库，前端只负责展示与提交 |

## 3. 技术栈全景

### 3.1 核心框架

```
Next.js 14（App Router）
├── React 18              — UI 渲染引擎，Server/Client Component 分离
├── TypeScript            — 全量类型覆盖（strict 模式）
└── Tailwind CSS 3        — 原子化样式 + 自定义 CSS 变量霓虹主题
```

### 3.2 完整技术栈矩阵

| 分类 | 技术/库 | 用途 |
|------|---------|------|
| **框架** | Next.js 14（App Router）| SPA 路由、SSR/CSR 混合渲染 |
| **语言** | TypeScript | 全量类型覆盖 |
| **样式** | Tailwind CSS + 自定义 CSS 变量 | 原子样式 + 霓虹主题系统 |
| **字体** | Advent Pro + Share Tech Mono | 标题字体 + 等宽代码字体 |
| **图标** | lucide-react | 统一图标库 |
| **HTTP 客户端** | 原生 Fetch API（封装于 `lib/api.ts`）| 所有 HTTP + 401/超时/断链拦截 |
| **状态管理** | React `useState` / `useReducer` | 组件级轻量状态 |
| **全局状态** | React Context（`NetworkContext`）| 网络断链全局通知 |
| **i18n** | 自研极简方案（`lib/i18n.ts` + `hooks/useLanguage.ts`）| 中英双语无刷新切换 |
| **类名工具** | `clsx` + `tailwind-merge`（封装为 `cn()`）| 动态类名安全合并 |
| **日期格式化** | 自研 `formatDate()`（`lib/utils.ts`）| 兼容 SQLite 多种时间格式 |
| **Canvas 动效** | 原生 Canvas API（`CyberParticles.tsx`）| 背景粒子特效 |

### 3.3 技术选型说明

**为什么不用状态管理库（Zustand/Redux）？**  
当前前端交互以「触发任务 + 读取列表」为主，没有复杂的跨页面共享状态，React 原生 useState + Context 足够。待后续功能复杂化时可平滑引入。

**为什么 i18n 不用 next-intl / react-intl？**
- 第三方库带来 50KB+ 运行时体积增量
- 需要路由重写、语言检测中间件、SSR 适配等复杂配置
- 当前只需中英双语，无复数规则、无日期本地化需求
- 自研方案总代码量 < 100 行，完全可控

## 6. 应用初始化与根布局

### 6.1 根布局 `app/layout.tsx`

根布局挂载全局字体、CSS 元数据，并包裹 `ClientShell`：

- **字体加载**：`Advent Pro`（标题/导航）、`Share Tech Mono`（代码/等宽数据）
- **全局 CSS**：引入 `globals.css`，注入 CSS 变量霓虹主题
- **元数据**：`title`、`description`、`viewport` 等 Next.js Metadata API
- **唯一子节点**：`<ClientShell>{children}</ClientShell>`

根布局保持为 Server Component，不含任何 `'use client'` 指令，确保 Next.js 静态优化不被污染。

### 6.2 客户端挂载壳 `ClientShell.tsx`

`ClientShell` 是前端的「客户端宇宙大爆炸点」，将所有需要 `'use client'` 的 Provider 与全局组件集中于此：

```
ClientShell（'use client'）
  └─► NetworkProvider           # 1. 最外层：全局网络断链状态
        ├─► CyberParticles       # 2. 背景粒子（fixed, z-0）
        ├─► AuthGuard            # 3. 鉴权守卫
        │     ├─► {children}     # 4. 页面内容
        │     └─► AiSidebar      # 5. AI 侧边栏（悬浮）
        └─► NeuralLinkAlert      # 6. 断链全局弹窗（z-[9999]）
```

**设计意图**：避免在 Server Component 根布局中混入客户端代码，保持 hydration 边界清晰，所有 Provider 的注册与清理均在 `ClientShell` 生命周期内完成。

## 8. 认证与鉴权体系

### 8.1 鉴权流程

```
用户访问任意页面
  └─► AuthGuard.tsx
        ├─ 路径为 /auth/login → 直接放行
        ├─ api.authStatus() → 系统未初始化 → 跳转 /auth/login（初始化模式）
        ├─ localStorage.token 不存在 → 跳转 /auth/login（登录模式）
        └─ 通过 → 渲染 {children} + AiSidebar
```

### 8.2 登录页双模式

`app/auth/login/page.tsx` 在挂载时调用 `api.authStatus()` 自动判断模式：

| 模式 | 触发条件 | 表单内容 |
|------|----------|----------|
| **初始化模式** | 系统从未设置管理员 | 创建管理员账号（用户名 + 密码 + 确认密码）|
| **登录模式** | 系统已有管理员 | 标准登录（用户名 + 密码）|

**JWT 存储**：登录成功后 JWT 写入 `localStorage.token`，后续请求由 `lib/api.ts` 的 `getHeaders()` 自动附加。

### 8.3 401 自动清退

`secureFetch()` 拦截所有 401 响应：

```
401 响应
  → localStorage.removeItem('token')
  → window.location.href = '/auth/login'
```

无需组件层感知，全局静默处理。

## 9. 数据获取层（API Client）

### 9.1 核心设计

`lib/api.ts` 是前端**唯一合法的 HTTP 出口**，所有组件禁止绕过此层直接调用 `fetch`。

### 9.2 `secureFetch` 核心拦截器

```
secureFetch(url, options)
  ├─ AbortController 15s 超时
  ├─ 超时 /网络错误 → notifyLinkDown() → NeuralLinkAlert 弹窗
  ├─ 401 → 清除 token → 跳转 /auth/login
  └─ 5xx → notifyLinkDown()
```

### 9.3 API 方法全览

#### 鉴权（不经 secureFetch，无需 JWT）

| 方法 | HTTP | 端点 |
|------|------|------|
| `authStatus()` | GET | `/auth/status` |
| `initAuth(username, password)` | POST | `/auth/init` |
| `login(username, password)` | POST | `/auth/login` |

#### 任务管理

| 方法 | HTTP | 端点 | 说明 |
|------|------|------|------|
| `getTasks(params)` | GET | `/tasks` | 支持分页/过滤参数 |
| `deleteTask(id)` | DELETE | `/tasks/{id}` | 单条删除 |
| `deleteBatchTasks(ids)` | POST | `/tasks/delete_batch` | 批量删除 |
| `purgeAllTasks()` | POST | `/tasks/purge` | 清空（请求体：`{confirm:'CONFIRM'}`）|
| `retryTask(id)` | POST | `/tasks/{id}/retry` | 失败任务重试 |

#### 触发后台任务

| 方法 | HTTP | 端点 |
|------|------|------|
| `triggerScan()` | POST | `/tasks/scan` |
| `triggerScrapeAll()` | POST | `/tasks/scrape_all` |
| `triggerFindSubtitles()` | POST | `/tasks/find_subtitles` |

#### 配置管理

| 方法 | HTTP | 端点 | 说明 |
|------|------|------|------|
| `getSettings()` | GET | `/tasks/settings` | 拉取完整配置 |
| `updateSettings(config)` | POST | `/tasks/settings` | 提交配置变更 |
| `resetSettings(target)` | POST | `/tasks/settings/reset` | target: 'ai' 或 'regex' |

#### 系统与 AI

| 方法 | HTTP | 端点 |
|------|------|------|

## 10. 状态管理体系

### 10.1 状态分层原则

| 层级 | 方案 | 适用场景 |
|------|------|----------|
| **组件局部状态** | useState / useReducer | 单组件内的 UI 状态（加载、分页、搜索词等）|
| **父子共享状态** | Props 下传 + 回调上传 | 容器组件持有，展示组件消费 |
| **全局横切状态** | React Context | 网络断链（NetworkContext）|
| **持久化状态** | localStorage | JWT token、ui_lang 语言偏好 |

### 10.2 NetworkContext — 全局断链状态

context/NetworkContext.tsx 提供全局网络状态，采用「window 桥接」模式解决 lib/api.ts（普通模块）与 React Context 之间的通信问题：

```
NetworkProvider 挂载
  → 注册 window.__setLinkDown = setIsLinkDown

api.ts secureFetch() 检测到网络异常
  → notifyLinkDown()
  → window.__setLinkDown?.(true)
  → NetworkContext.isLinkDown = true
  → NeuralLinkAlert 弹窗显示

NetworkProvider 卸载
  → delete window.__setLinkDown   # 自动清理，无内存泄漏
```

### 10.3 useLanguage Hook — 语言状态

hooks/useLanguage.ts 提供语言状态管理，返回 { lang, setLang, t }：

```
setLang(newLang)
  → localStorage.setItem('ui_lang', newLang)
  → dispatchEvent(new CustomEvent('languageChanged'))

其他组件中的 useLanguage()
  → addEventListener('languageChanged', handler)
  → handler 读取 localStorage 更新 lang
  → 组件重新渲染（t() 返回新语言文本）
  → 100ms 后 window.location.reload() 确保完整刷新
```

### 10.4 SettingsHub — 配置状态持有者

SettingsHub 是设置模块的唯一状态持有者，采用「集中持有，props 下发」模式：

```
SettingsHub
  → api.getSettings() → config（mount 时）
  → config 通过 props 传给所有子 Tab 组件
  → 子组件通过回调触发 setConfig()（局部更新）
  → 点「保存」→ 1+1 路径约束校验 → api.updateSettings(config)
```

## 14. AI 交互前端舱

### 14.1 组件定位

### 14.2 核心状态

| 状态 | 说明 |
|------|------|
| `isOpen` | 侧边栏展开/收起 |
| `messages` | 对话历史记录 |
| `loading` | 等待 AI 响应中 |
| `selectedMsgIdx` | 已封死的候选消息 idx 集合 |
| `visibleIdx` | 已完成入场动画的消息 idx 集合 |

### 14.3 候选列表解析（`__CANDIDATES__`）

后端 DOWNLOAD 意图返回模糊结果时，响应文本末尾附加结构化数据：

```
找到「X」的多个版本，请选择：__CANDIDATES__["片名 (年份)", ...]
```

`parseCandidates(content)` 以 `__CANDIDATES__` 为分隔符切割：
- `text`：引导语（后端本地拼接，不走 LLM，前端直接展示）
- `candidates`：JSON 解析后的字符串数组，渲染为快捷选择按钮

### 14.4 候选封死机制

- 只有**最后一条含候选的消息**按钮处于激活状态（`lastCandidateIdx` 计算）
- 任何新消息发送时，立即封死所有含候选消息（加入 `selectedMsgIdx`），防止二次选择
- 已封死按钮：`disabled`、变灰、`cursor: not-allowed`，硬阻断点击
- 候选按钮点击 + `handleSendText` 封死逻辑双重保险，防止重复触发

### 14.5 动作指令执行

| action 值 | 前端行为 | i18n key |
|-----------|----------|----------|
| `ACTION_SCAN` | api.triggerScan() | ai_scan_triggered |
| `ACTION_SCRAPE` | api.triggerScrapeAll() | ai_scrape_triggered |
| `ACTION_SUBTITLE` | api.triggerFindSubtitles() | ai_subtitle_triggered |

DOWNLOAD action 由后端 agent 内部完成，前端不处理。每个 action 执行后均在对话中追加国际化提示消息。

## 15. 通用基础设施组件

### 15.1 AuthGuard

路由鉴权守卫，检查流程：

1. 路径为 `/auth/login` → 直接放行
2. `api.authStatus()` 返回未初始化 → 跳转 `/auth/login`（初始化模式）

### 15.4 CyberParticles

## 19. 错误处理体系

### 19.1 三层防线架构

```
第一层：lib/apiError.ts
  → ApiError 结构化错误类
  → notifyLinkDown() 触发网络断链通知

第二层：NeuralLinkAlert
  → 捕获所有 5xx / 超时 / 网络中断
  → 全屏覆盖弹窗，提示用户重连
  → 用户主动触发 reload() 恢复

第三层：app/error.tsx
  → Next.js App Router Error Boundary
  → 捕获渲染阶段未处理异常
  → 赛博朋克风格错误页 + 重试按钮
```

### 19.2 ApiError 错误码

| code | 触发场景 | status |
|------|----------|--------|
| `REQUEST_TIMEOUT` | 15s 超时 | 408 |
| `NETWORK_ERROR` | 网络不可达 | 0 |
| `UNAUTHORIZED` | 401 响应 | 401 |
| `SERVER_ERROR` | 5xx 响应 | 500+ |
| `INVALID_CONTENT` | 非 JSON 响应 | — |

### 19.3 401 自动清退

secureFetch 拦截 401 时自动清除 localStorage.token 并跳转 /auth/login，无需组件层感知。

## 20. 网络与代理机制

### 20.1 Next.js 反向代理

开发/生产环境均通过 `next.config.js` rewrites 将 `/api/v1/*` 代理到后端服务，前端代码中无需关心后端实际地址。

### 20.2 超时与重试策略

- 所有请求统一 15s 超时（`AbortController`）
- 无自动重试（避免重复触发后台任务）
- 超时后触发断链弹窗，由用户手动重连

### 20.3 图片代理

`SecureImage` 组件将本地文件路径转为 `/api/v1/public/image?path=...`，后端验证路径安全性后返回图片二进制流。前端禁止直接拼接文件系统路径。

| i18n | 所有 UI 字符串通过 `t(key)` 获取，禁止硬编码 |
| 新增 i18n key | 必须同时在 zh 和 en 字典中添加 |

### 21.3 构建与部署

```bash
npm run build   # 生产构建
npm run start   # 启动生产服务器
```

Docker 部署参见 `docs/01_架构设计/06_AIO全栈部署指南_V1.0.md`。

## 22. 架构约束与红线清单

以下为前端架构强制约束，违反即视为架构破坏性变更，需经架构评审：

### 22.1 数据层红线

| 红线 | 说明 |
|------|------|
| 禁止裸 fetch | 所有 HTTP 调用必须经过 `lib/api.ts` |
| 禁止硬编码类型 | Task.status / sub_status 必须从 `types/index.ts` 引用 |
| 禁止硬编码正则 | 文件名清洗正则必须来自后端 `filename_clean_regex` 字段 |
| 禁止前端存业务规则 | 正则规则、AI 规则来自后端数据库，前端只展示与提交 |

### 22.2 组件层红线

| 红线 | 说明 |
|------|------|
| 禁止巨石组件 | 单文件组件超过 400 行时必须拆分 |
| 禁止硬编码字符串 | 所有 UI 文本通过 `t(key)` 获取 |
| 禁止离散色值 | 颜色统一使用 CSS 变量，禁止 `#00e6f6` 等离散值出现在组件中 |
| 禁止 Server Component 混入 client 代码 | `app/layout.tsx` 必须保持为纯 Server Component |

### 22.3 路径约束红线

| 红线 | 说明 |
|------|------|
| 1+1 媒体库约束 | 电影媒体库启用数 ≤ 1，剧集媒体库启用数 ≤ 1 |
| 两类型必须同时存在 | 有媒体库时 movie 和 tv 两种类型必须同时启用 |
| 图片路径必须代理 | 所有本地图片路径必须经 `SecureImage` 组件代理，禁止直接拼接 |

### 22.4 i18n 红线

| 红线 | 说明 |
|------|------|
| 新增 key 必须双语 | zh 和 en 字典必须同步添加，否则 TypeScript 报错 |
| 禁止运行时字符串拼接 | i18n key 必须是字面量，禁止动态拼接 key 字符串 |
