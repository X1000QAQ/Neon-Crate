## 一、系统定位与设计哲学

Neon Crate 前端是「数据容器编排引擎的工业级霓虹控制面板」。

| 原则 | 具体体现 |
|------|----------|
| **类型即契约** | `types/index.ts` 唯一类型来源，与后端 Pydantic 严格对齐 |
| **单一数据入口** | 所有 HTTP 调用只能经过 `lib/api.ts`，禁止裸 `fetch` |
| **国际化优先** | 所有 UI 字符串通过 `t(key)` 获取，严禁硬编码 |
| **错误防线纵深** | `apiError.ts` → `NeuralLinkAlert` → `error.tsx` 三层递进 |
| **前端不存业务规则** | 正则/AI 规则来自后端数据库，前端只展示与提交 |

## 五、核级重构弹窗（RebuildDialog.tsx）— v1.0.0

### 开门重置器（防止幽灵记忆）

**业务链路**：
```
1. 监听 open 状态与 task.id 变化 -> 
2. 触发弹窗展示 -> 
3. 彻底清空上一轮的幽灵记忆 (keyword, results, selected) -> 
4. 重新灌入当前 task 的基础参数
```

**实现**：
```typescript
useEffect(() => {
  if (open) {
    setMediaType(task.media_type || 'movie');
    setKeyword(task.title || '');
    setResults([]);
    setSelected(null);
    setSearchErr(null);
    setSeason(task.season ?? 1);
    setEpisode(task.episode ?? '');
  }
}, [open, task.id]);
```

**防止 Dialog 幽灵记忆陷阱**：
- `if (!open) return null` 只隐藏不卸载，导致状态被永久冻结
- 此 useEffect 确保每次打开新任务时，状态都被重置

### 核级重构执行流程

**业务链路**：
```
1. 用户点击「执行核级重构」按钮 -> 
2. 验证 tmdb_id 非空 -> 
3. 构建请求参数（task_id, is_archive, nuclear_reset=true）-> 
4. 调用 api.rebuildTask()
  └─► POST /tasks/manual_rebuild
        └─► 后端处理（见 @docs/02_数据契约/02_API规范与鉴权.md）
              └─► 返回重建结果 {success, rebuilt: {nfo, poster, subtitle, nuclear}}
                    └─► 前端显示成功提示 + 刷新列表
```

### 搜索业务链路

**业务链路**：
```
1. 用户输入关键词 -> 
2. 防抖处理（500ms）-> 
3. 调用 api.searchTmdb()
  └─► POST /tasks/search_tmdb
        └─► 返回候选列表 [{tmdb_id, title, year, ...}, ...]
```

## 六、双轨路径显示（UniversalMediaRow / MediaTable.tsx）— V16 等高扁平折叠列表

> **[v1.0.0 重构]** `TaskRow` 已废除。双轨路径展示逻辑现统一在 `UniversalMediaRow` 中实现，适用于电影（level=0）和单集（level=2）节点（有 `task` 对象时显示）。剧集根和季节点无 `task` 对象，不显示路径信息。

### 路径设计

**前世（SRC）**：`file_path`
- 含义：文件在下载源中的原始路径
- 示例：`/downloads/The.Matrix.1999.1080p.BluRay.x264.mkv`
- 用途：追溯文件来源、重新扫描、调试

**今生（DST）**：`target_path`
- 含义：文件入库后的目标路径（已重命名、已移动）
- 示例：`/media/movies/The Matrix (1999)/The Matrix (1999).mkv`
- 用途：媒体库寻址、播放器调用、元数据关联

### 前端 UI 展示

**业务链路**：
```
1. 检查 task.target_path 是否存在 -> 
2. 若存在则显示为"入库路径"（置于上方）-> 
3. 使用 t('path_dst') 获取多语言标签 -> 
4. 显示 task.file_path（下载源中的原始路径）-> 
5. 使用 t('path_src') 获取多语言标签 -> 
6. 用于追溯文件来源
```

**实现**：
```typescript
// 🚀 今生：目标入库路径 (置于上方)
{task.target_path && (
  <p className="text-cyber-cyan/30 text-xs truncate mt-1" title={task.target_path}>
    <span className="text-cyber-cyan/50 font-mono mr-1">{t('path_dst')}:</span>
    {task.target_path}
  </p>
)}

// 前世：原始绝对路径 (置于下方)
<p className="text-cyber-cyan/40 text-xs truncate mt-0.5" title={task.file_path}>
  <span className="text-cyber-cyan/60 font-mono mr-1">{t('path_src')}:</span>
  {task.file_path}
</p>
```

### NetworkContext — 全局断链状态

```

```

### useNeuralLinkStatus — 全局单例“量子态枢纽”（Neural Link）

**定位**：统一驱动全站 Holographic Void 状态栏（`量子态 / 神经链路`），杜绝组件各自轮询造成的网络压力与状态分裂。

**设计模式**：模块级 Singleton Store + 订阅者模式（observer）。

- **全站单轮询器**：无论多少组件订阅（`AiSidebar/MiniLog/SystemMonitor/...`），底层只会启动一个轮询器；最后一个订阅者卸载后自动停表。
- **真实探活**：以 `/system/stats` 作为轻量探活锚点；结合 `neon-network-up/down` 事件（由 `lib/api.ts` 统一派发）快速切换链路态。
- **后台态聚合**：轮询读取 `scan/scrape/subtitle` 的 `/status`，聚合为 `quantum_state`（stable/syncing/processing/degraded）。

**输出契约**（Hook 只返回枚举，不做字符串拼接；i18n 由 UI 层负责）：
- `quantum_state`: `stable | syncing | processing | degraded`
- `neural_link`: `active | offline | probing`
- `updated_at`: ms epoch

## 十四、高性能渲染范式（装甲化法典 Phase 1-2）

> **[2026-03-16 装甲化]** 除垢行动 Phase 1-3 完成后，以下规范正式上升为架构铁律，违反任一条款视为破坏性变更。

### 法则 R-01：DOM 直操防重绘（NeuralWaveform 标准）

**场景**：60fps 高频动画、SVG 路径插值、Canvas 粒子等实时更新场景。

**铁律**：
- **绝对禁止**将动画帧数据绑定到 `useState`，每帧 `setState` 将触发全组件重绘，100% 造成帧率崩溃。
- **必须**采用 `useRef` 持有 DOM 节点，通过 `element.setAttribute()` / `element.style` 直接操作，完全绕过 React 渲染管线。

**标准实现**（参考 `NeuralWaveform.tsx`）：
```typescript
// ❌ 禁止：每帧 setState → 全组件重渲染
const [waveAmplitude, setWaveAmplitude] = useState(0);
useEffect(() => {
  const id = setInterval(() => setWaveAmplitude(t => t + 0.05), 16);
  return () => clearInterval(id);
}, []);

// ✅ 正确：useRef 直接操作 SVG DOM，零 React 开销
const path1Ref = useRef<SVGPathElement>(null);
useEffect(() => {
  let frame = 0;
  const animate = () => {
    frame += 0.05;
    path1Ref.current?.setAttribute('d', buildPath(frame));
    rafRef.current = requestAnimationFrame(animate);
  };
  rafRef.current = requestAnimationFrame(animate);
  return () => { if (rafRef.current) cancelAnimationFrame(rafRef.current); };
}, []);
```

### 法则 R-02：Context 渲染风暴熔断（APISettings 标准）

**场景**：表单输入框直接绑定 `Context.updateSetting`，每次击键触发全局 Context 重绘。

**铁律**：所有写入全局 Context 的输入框，必须严格执行「三步走」缓冲策略（**Local State Buffer + onBlur Sync 架构**）：

```
[Step 1] onChange → 仅更新本地 localState（阻断 Context 重绘）
[Step 2] onBlur  → 调用 updateSetting 写入全局 Context（批量提交）
[Step 3] 读取时  → localState 优先，Context 兜底（保证输入中的响应性）
```

> **适用范围**：所有高频输入/粘贴组件（包括 API Key 输入框、AI 人格编辑器、正则规则编辑器等）。
> 违反此规则将导致 React 合成事件回收（SyntheticEvent pooling）引发渲染崩溃，症状为粘贴后光标跳位或内容截断。

**标准实现**（参考 `APISettings.tsx` / `InferenceSettings.tsx`）：
```typescript
// 本地缓冲 state，键盘输入仅在本地流转
const [localValues, setLocalValues] = useState<Partial<Record<FieldKey, string>>>({});

const getVal = (key: FieldKey) =>
  key in localValues ? localValues[key]! : (config.settings[key] ?? '');

<input
  value={getVal('tmdb_api_key')}
  onChange={(e) => setLocalValues(p => ({ ...p, tmdb_api_key: e.target.value }))}
  onBlur={(e) => updateSetting('tmdb_api_key', e.target.value)}  // ← 失焦才提交
/>
```

**适用范围**：Settings 模块所有文本输入框。开关类 `<NeuralCoreSwitch>` 无高频触发风险，可直接 `onChange` 写入。

## 十五、SPA 尊严与异步安全（装甲化法典 Phase 1-3）

### 法则 S-01：SPA 绝对禁忌

| 禁止行为 | 危害 | 替代方案 |
|----------|------|----------|
| `alert(message)` | 阻塞 JS 主线程，UX 破坏性极强 | 局部/全局 Toast 组件 |
| `window.location.reload()` | 销毁前端内存树，重置所有状态 | 局部 `setState` 刷新目标数据 |
| `window.location.href = '...'` | 同上，除非是登出/鉴权跳转 | Next.js `router.push()` |

**Toast 标准实现**（参考 `SettingsHub.tsx` / `MediaWall.tsx`）：
```typescript
const [toast, setToast] = useState<string | null>(null);
const toastTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null);

const showToast = useCallback((msg: string) => {
  if (toastTimerRef.current) clearTimeout(toastTimerRef.current);  // 防抖：清除旧计时器
  setToast(msg);
  toastTimerRef.current = setTimeout(() => setToast(null), 3000);  // 3s 自动消失
}, []);

// 卸载清理（防内存泄漏）：
useEffect(() => { return () => { if (toastTimerRef.current) clearTimeout(toastTimerRef.current); }; }, []);
```

### 法则 S-02：防爆轮询机制（递归 setTimeout 标准）

**铁律**：全站禁止以 `setInterval` 进行数据轮询，必须采用「递归 `setTimeout` + 指数退避」模式。

**`setInterval` 的死亡陷阱**：
- 上一次请求未完成时，下一次定时器已触发 → **请求堆积、响应乱序**
- 组件卸载时若未精确 `clearInterval` → **僵尸定时器持续运行、内存泄漏**
- 嵌套 `clearInterval` + 重建 `setInterval` 的「变相指数退避」实现 → **逻辑混乱，极易漏清理**

**标准实现**（参考 `StatsOverview.tsx`）：
```typescript
const scanBoostTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null);

const pollStats = async (startedAt: number) => {
  const elapsed = Date.now() - startedAt;
  if (elapsed > 30000) return;  // 30s 后自然退出，零泄漏

  await loadStats();            // 等待当前请求完成，天然防堆积

  const nextDelay = elapsed < 10000 ? 1500 : 5000;  // 指数退避
  scanBoostTimerRef.current = setTimeout(() => pollStats(startedAt), nextDelay);
};

// 触发时：
if (scanBoostTimerRef.current) clearTimeout(scanBoostTimerRef.current);
void pollStats(Date.now());

// 卸载清理：
useEffect(() => {
  return () => { if (scanBoostTimerRef.current) clearTimeout(scanBoostTimerRef.current); };
}, []);
```

**三阶段退避模型**：

| 阶段 | 时间范围 | 间隔 | 目的 |
|------|----------|------|------|
| 极速响应 | 0 – 10s | 1.5s | 快速感知操作结果 |
| 平稳监控 | 10 – 30s | 5s | 降低磁盘/网络压力 |
| 休眠保护 | 30s 后 | 停止 | 零资源占用 |

### 法则 S-03：请求信号链路（AbortController 标准）

**铁律**：所有可被中断的 Fetch 请求必须桥接 `AbortController.signal`，确保组件卸载或操作取消时正确丢弃幽灵请求。

**完整链路**（参考 `AiSidebar.tsx` + `lib/api.ts`）：
```typescript
// 前端：每次发送前中止上一次飞行请求
abortControllerRef.current?.abort();
abortControllerRef.current = new AbortController();
const res = await api.chat(text, abortControllerRef.current.signal);

// api.ts：外部 signal 桥接到内部超时 controller
if (options?.signal) {
  options.signal.addEventListener('abort', () => controller.abort());
}

// catch 中静默处理 AbortError（用户主动取消，非异常）：
if (e instanceof Error && e.name === 'AbortError') return;
```

**组件卸载时的清理标准**：
```typescript
useEffect(() => {
  return () => { abortControllerRef.current?.abort(); };
}, []);
```

### 法则 I-01：i18n 绝对覆盖（零硬编码中文）

**铁律**：全站所有 UI 可见字符串，无论场景（正常流程、loading 状态、error catch、占位符），**100% 必须**通过 `t('key')` 宏调用获取，绝对禁止任何形式的硬编码中文字符串。

**高危场景强制 i18n 清单**：

| 场景 | ❌ 禁止 | ✅ 正确 |
|------|---------|--------|
| 按钮 loading 状态 | `{resetting ? '重置中...' : ...}` | `{resetting ? t('btn_resetting') : ...}` |
| 按钮 executing 状态 | `'执行中...'` | `t('overlay_btn_executing')` |
| Error catch 兜底 | `error.message \|\| '未知错误'` | `error.message \|\| t('error_unknown')` |
| 空状态占位 | `'暂无简介'` | `t('text_no_overview')` |
| 确认弹窗文本 | `'该资源已存在'` | `t('overlay_duplicate_warning')` |

**i18n key 命名规范**：
- `btn_*`：按钮文本（含状态变体，如 `btn_saving`, `btn_resetting`）
- `error_*`：错误信息（如 `error_unknown`）
- `overlay_*`：模态框/覆层文本
- `msg_*`：操作反馈消息

**双语同步原则**：`i18n.ts` 中 `zh` 和 `en` 字典必须在同一次提交中同步添加键值，禁止单语种提交。

#### 🚨 动态键值保护红线（DO NOT DELETE DYNAMIC KEYS）

前端存在领域核心的“动态拼接翻译键”，典型形式如下：

```ts
t('status_' + task.status)
t('sub_status_' + task.sub_status)
t('ui_' + dynamicKey)
```

因此，任何以 **`status_` / `sub_status_` / `ui_`** 开头的 i18n 词条都属于 **领域核心动态键**：
- **静态搜索命中可能为 0**（因为 key 在运行时拼接），但依然是“在用”的。
- **绝对禁止**被静态扫描工具或 AI 清理误删。
- 字典瘦身时必须将这些前缀整体加入保护名单，优先保证语义正确与前后端契约一致性。
