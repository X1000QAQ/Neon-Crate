## 一、系统定位

Neon Crate 是面向 NAS / 家庭媒体服务器场景的**全自动影音库管理引擎**。核心能力：

- **AI 总控神经**：自然语言指令 → LLM 意图路由 → 自动触发扫描/刮削/下载/字幕任务
- **物理文件编排**：扫描下载目录 → 清洗文件名 → TMDB 刮削元数据 → SmartLink 硬链接归档
- **数据持久化**：SQLite WAL 模式，管理所有媒体任务状态、配置、归档记录
- **安全网关**：JWT 无状态鉴权 + Fernet 加密配置
- **AIO 静态服务**：生产模式单端口、单进程托管前后端

## 四、核心数据流与业务链路

### 4.1 全量归档链路（v1.0.0）

**业务链路**：1. 防重锁获取 → 2. API Key 校验 → 3. 任务列表获取 → 4. NFO 短路拦截 → 5. AI 提炼 → 6. TMDB 搜索 → 7. 防重拦截 → 8. 就地补录/归档全链路 → 9. 字幕白嫖

```
POST /scan
  └─► ScanEngine（目录遍历 + 格式/大小过滤）
        └─► MediaCleaner（数据库驱动正则清洗）
              └─► db.insert_task()（path UNIQUE 防重入库）

POST /scrape_all
  └─► 获取防重锁（非阻塞，blocking=False）
        └─► 校验 TMDB API Key
              └─► 获取待刮削任务（双表联查：tasks + media_archive）
                    └─► 逐个处理任务
                          ├─► NFO 短路拦截（本地 NFO 存在）
                          │     └─► 解析 NFO → 提取 TMDB/IMDb ID → 跳过 AI/TMDB
                          ├─► AI 提炼逻辑（NFO 不存在）
                          │     └─► 正则清洗 → AI Agent 分析 → 提取查询词/年份/类型
                          ├─► TMDB 搜索与防重拦截
                          │     └─► 搜索 TMDB → 精确/宽松匹配 → 获得 TMDB/IMDb ID
                          ├─► IMDb 金标准防重检测
                          │     └─► check_media_exists() → 若重复标记 ignored
                          ├─► 就地补录（文件在 library 路径）
                          │     └─► 补充 NFO/海报 → 不移动文件
                          ├─► 归档全链路（文件在下载目录）
                          │     └─► SmartLink.create_link() → 移动文件 → 写入元数据
                          └─► 字幕白嫖（本地字幕检测）
                                └─► 发现本地字幕 → 标记 success；否则标记 pending

POST /find_subtitles
  └─► SubtitleEngine
        ├─► 本地字幕检测（已有则跳过，零 API 消耗）
        └─► OpenSubtitles API → {视频名}.ai.zh-CN.srt
```

**关键设计**：
- **防重锁**：`threading.Lock(blocking=False)` 非阻塞获取，失败立即返回，不排队
- **双表联查**：`get_tasks_needing_scrape()` 同时查询 `tasks`（热表）和 `media_archive`（冷表），按 `target_path` 去重
- **NFO 短路**：若本地存在 NFO 文件，直接解析提取元数据，跳过 AI 和 TMDB 搜索；解析器具备 **errors=replace + 生化清洗 + 正则兜底** 三层装甲，对第三方脏 NFO 具备极强生存能力
- **AI 提炼**：调用 `AIAgent.ai_identify_media()` 进行文件名语义分析，AI 输出 `filename_year` 和 `knowledge_year` 两份年份证据
- **防重拦截**：`check_media_exists()` 精确到季集号，防止同剧不同集被误杀
- **就地补录**：文件已在媒体库，仅补充元数据，不移动文件
- **归档全链路**：文件在下载目录，执行 SmartLink（硬链接 > 软链接 > 复制）、重命名、写入 NFO/海报
- **字幕白嫖**：归档完成后立即检测本地字幕，若有则标记 `success`，零 API 消耗

### 4.4 并发防重链路（V2.0 双重护盾加固）

```
每个任务模块独立的 threading.Lock + 状态快照（dict）
  1. 非阻塞获取锁（blocking=False）
  -> 2. 失败 → 立即返回，丢弃冗余请求，不排队不等待
  -> 3. 成功 → 锁内二次状态复检（Issue 4.1 物理修复）
        scrape_all_status["is_running"] 仍为 True？
        → 主动释放锁并退出，消除极端竞态窗口
  -> 4. 通过二次复检 → 写入 scrape_all_status["is_running"]=True（并清空 error）
  -> 5. 执行任务，finally 块无条件：
        - scrape_all_status["is_running"]=False（解除前端“运行中”锁定）
        - 释放 threading.Lock（系统自愈）
```

**状态快照字段约定**（`_shared.py` / tasks/_shared.py）：
```python
scrape_all_status["is_running"]      # bool: 运行态
scrape_all_status["error"]           # Optional[str]: 最近一次错误
scrape_all_status["processed_count"] # int: 最近一次处理数量
scrape_all_status["last_run_time"]   # float: 最近一次运行时间戳（epoch seconds）
```

### 4.5 SQLite WAL 模式与线程级连接池（v1.0.0）

**WAL 模式设计**：
- **启用方式**：`PRAGMA journal_mode=WAL`（在 `_get_conn()` 中自动执行）
- **优势**：
  - 读不阻塞写：多个读线程可并发执行，不等待写操作完成
  - 写不阻塞读：写操作在 WAL 文件中进行，不锁定主数据库
  - 原子性保证：事务提交时自动 checkpoint，确保数据一致性
- **文件结构**：
  ```
  media_database.db      # 主数据库文件
  media_database.db-wal  # 预写日志（Write-Ahead Log）
  media_database.db-shm  # 共享内存（Shared Memory）
  ```

**线程级连接池复用机制**：
```python
# db_manager.py: _get_conn() 实现
def _get_conn(self) -> sqlite3.Connection:
    """
    获取当前线程的持久化 SQLite 连接（线程级连接池核心方法）
    
    业务链路：
    1. 检查当前线程是否已有缓存连接 -> 
    2. 若有则直接复用（零开销）-> 
    3. 若无则创建新连接 -> 
    4. 配置 WAL 模式和行工厂 -> 
    5. 缓存到线程本地存储 -> 
    6. 返回连接
    """
    conn = getattr(self._local, "conn", None)
    if conn is None:
        conn = sqlite3.connect(self.db_path, check_same_thread=False)
        conn.execute("PRAGMA journal_mode=WAL")
        conn.row_factory = sqlite3.Row
        self._local.conn = conn
    return conn
```

**性能优化**：
- **消除高频 connect/close 开销**：每次查询都要建立连接的性能灾难已消除
- **线程安全**：每个线程独立连接，无竞态条件，由 `db_lock` 保护
- **自动连接复用**：同一线程的多次查询共享一个连接，减少系统调用

## 六、设计哲学

| 原则 | 具体体现 |
|------|----------|
| **数据库即真相源** | 所有正则规则、AI 规则均存储于 SQLite，严禁硬编码 |
| **默认拦截，显式豁免** | 所有 `/api/v1/*` 路由默认要求 JWT |
| **零依赖降级** | 各服务步骤独立容错，单步失败不崩溃整个任务 |
| **极简部署** | AIO 模式单进程、单端口，适合 NAS 场景 |
| **防重叠调度** | threading.Lock 防止同类任务并发执行 |

| `lib/i18n.ts` | 中英双语翻译字典（自研，< 100 行）|
| `types/index.ts` | 前端唯一类型来源，与后端 Pydantic 严格对齐 |
| `context/` | 全局状态（NetworkContext 断链通知）|

## 六、设计哲学

| 原则 | 具体体现 |
|------|----------|
| **数据库即真相源** | 所有正则规则、AI 规则均存储于 SQLite，严禁硬编码 |
| **默认拦截，显式豁免** | 所有 `/api/v1/*` 路由默认要求 JWT |
| **零依赖降级** | 各服务步骤独立容错，单步失败不崩溃整个任务 |
| **极简部署** | AIO 模式单进程、单端口，适合 NAS 场景 |
| **防重叠调度** | threading.Lock 防止同类任务并发执行 |
